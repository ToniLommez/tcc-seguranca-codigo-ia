VULNERABILITY #3
==============================================================================
CWE ID: CWE-22
Name: Path Traversal in Static File Serving
Severity: Critical
File: backend/main.cpp
Lines: 89-106
Description: The static file serving routes construct file paths by directly
concatenating user-supplied URL path segments with the FRONTEND_DIR base
directory. The regex patterns R"(/([^/].*\.(html|css|js|png|ico|svg)))" and
R"(/((?:[^/]+/)+[^/]+\.(html|css|js|png|ico|svg)))" do not filter out
directory traversal sequences such as "..". An attacker can craft a URL like
"/..%2F..%2Fetc%2Fpasswd.html" or "/../../../etc/passwd.html" (if the server
decodes the URL before matching) to read arbitrary files from the server's
filesystem, as long as the filename ends with one of the allowed extensions.
Exploitation: An attacker sends a request such as
GET /../../backend/config.h.html or GET /..%2f..%2fbackend/config.h (depending
on URL decoding behavior) to read sensitive source files, configuration, or
the Firebase credentials JSON file from the server filesystem.

VULNERABILITY #5
==============================================================================
CWE ID: CWE-326
Name: Insufficient PBKDF2 Iteration Count
Severity: High
File: backend/handlers/auth_handler.cpp
Line: 42
Description: The PBKDF2 password hashing uses only 10,000 iterations. Modern
security guidelines (OWASP 2023) recommend a minimum of 600,000 iterations
for PBKDF2-SHA256. With 10,000 iterations, password hashes can be brute-forced
at high speed using commodity GPUs.
Exploitation: If the database (Firebase Firestore or local SQLite) is
compromised, an attacker can crack user passwords efficiently using GPU-
accelerated brute-force attacks. At 10,000 iterations, millions of password
guesses per second are feasible on modern hardware.

VULNERABILITY #6
==============================================================================
CWE ID: CWE-208
Name: Timing-Vulnerable JWT Signature Comparison
Severity: High
File: backend/auth/jwt_helper.cpp
Line: 132
Description: The JWT signature verification uses a simple string comparison
operator (!=) to compare the expected signature with the provided signature:
"if (expected_sig_enc != sig_enc) return false;". Standard string comparison
is not constant-time and can leak information about how many characters of the
signature matched, enabling a timing side-channel attack.
Exploitation: An attacker can iteratively guess the correct HMAC signature
byte-by-byte by measuring response time differences. Each correct byte will
cause the comparison to take marginally longer before failing, allowing the
attacker to forge valid JWT tokens without knowing the secret key.

VULNERABILITY #8
==============================================================================
CWE ID: CWE-338
Name: Weak Pseudo-Random Number Generator for ID Generation
Severity: High
File: backend/handlers/music_handler.cpp
Lines: 30-36
Description: The generateId() function uses std::mt19937_64 seeded with the
current system time (std::chrono::system_clock::now().time_since_epoch().count()).
Mersenne Twister is not a cryptographically secure PRNG, and time-based seeding
makes the output predictable. An attacker who knows the approximate time a song
was uploaded can predict or enumerate possible song IDs.
Exploitation: An attacker can predict song IDs by knowing the approximate
upload time, then access or manipulate songs they should not have access to.
The use of only 24 hex characters from a predictable generator further reduces
the ID space.

Severity: Medium
File: backend/handlers/music_handler.cpp
Lines: 105-110
Description: The MP3 validation checks only the Content-Type header or the

VULNERABILITY #18
==============================================================================
CWE ID: CWE-352
Name: Missing CSRF Protection
Severity: Medium (noted, mitigated by JWT)
File: backend/main.cpp, backend/handlers/auth_handler.cpp
Lines: main.cpp:62-70, auth_handler.cpp:63-68
Description: The application has no CSRF protection mechanisms (no CSRF
tokens, no SameSite cookie attributes, no Origin/Referer validation). While
the API uses Bearer token authentication via the Authorization header (which
provides some inherent CSRF protection since browsers do not automatically
attach custom headers), the wildcard CORS policy (Vulnerability #4)
undermines this protection by allowing any origin to make requests with
custom headers.
Exploitation: Combined with the wildcard CORS policy, a malicious website
could use JavaScript to make authenticated API requests with the
Authorization header if the token is accessible (e.g., stored in
localStorage), effectively performing CSRF attacks to upload malicious
content or access user data.
