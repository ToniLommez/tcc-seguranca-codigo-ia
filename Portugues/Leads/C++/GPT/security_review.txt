VULNERABILITY #1
================================================================================
CWE ID:       CWE-798 (Use of Hard-coded Credentials)
Name:         Hardcoded JWT Secret in Source Code and Default Configuration
Severity:     Critical
File:         include/app_config.hpp, line 8
              src/app_config.cpp, line 21
Description:  The JWT signing secret is hardcoded as a default value directly in
              the source code: "change-this-secret-in-production-2026". If the
              config.json file does not contain a "jwt_secret" field, the
              application silently uses this well-known default secret.
Exploitation: An attacker who knows this default secret (which is visible in
              the source code) can forge valid JWT tokens for any user, gaining
              full unauthorized access to the application.

VULNERABILITY #2
================================================================================
CWE ID:       CWE-319 (Cleartext Transmission of Sensitive Information)
Name:         No TLS/HTTPS Support - Credentials Transmitted in Plaintext
Severity:     Critical
File:         src/server.cpp, line 758
Description:  The server uses plain HTTP (httplib::Server, not httplib::SSLServer)
              with no TLS configuration. Login credentials (email and password),
              JWT tokens in Authorization headers, and all lead data are
              transmitted in cleartext. The server also binds to 0.0.0.0.
Exploitation: Any attacker on the same network can perform a man-in-the-middle
              attack or passive network sniffing to capture user passwords,
              JWT tokens, and all lead data in transit.

VULNERABILITY #3
================================================================================
CWE ID:       CWE-307 (Improper Restriction of Excessive Authentication Attempts)
Name:         No Rate Limiting or Account Lockout on Login
Severity:     High
File:         src/server.cpp, lines 350-378
Description:  The login endpoint (/api/auth/login) has no rate limiting,
              account lockout mechanism, or progressive delay after failed
              attempts. The registration endpoint similarly has no rate limiting.
Exploitation: An attacker can perform unlimited brute-force password guessing
              attacks against any known email address. With the 6-character
              minimum password length and no lockout, weak passwords can be
              quickly cracked through online attacks.

VULNERABILITY #4
================================================================================
CWE ID:       CWE-79 (Cross-Site Scripting - Stored XSS)
Name:         No Output Encoding/Sanitization of User-Supplied Data
Severity:     High
File:         src/server.cpp, lines 48-65, 462, 507, 548, 626
Description:  User-supplied data (fullName, email, phone, company, jobTitle,
              source, notes, interaction note/type) is stored and returned in
              JSON responses without any sanitization or encoding. The fields
              fullName, company, notes, and interaction note accept arbitrary
              text with no character restrictions.
Exploitation: An attacker could register a lead with a fullName or notes field
              containing malicious JavaScript (e.g., <script>alert(1)</script>).
              When another user views this lead through a vulnerable frontend,
              the script executes in their browser context, potentially stealing
              their JWT token.

VULNERABILITY #5
================================================================================
CWE ID:       CWE-208 (Observable Timing Discrepancy)
Name:         Timing Side-Channel in JWT Signature Verification
Severity:     Medium
File:         src/security.cpp, line 314
Description:  The JWT signature verification at line 314 uses a standard string
              comparison operator (!=) to compare the expected and actual
              signatures: "if (expected != parts[2])". String comparison in
              C++ short-circuits on the first differing character, creating a
              timing oracle. (Note: password verification correctly uses a
              constant-time comparison.)
Exploitation: An attacker can repeatedly send JWTs with slightly varied
              signatures and measure response times to progressively determine
              the correct HMAC signature byte-by-byte, eventually forging a
              valid JWT without knowing the secret key.
