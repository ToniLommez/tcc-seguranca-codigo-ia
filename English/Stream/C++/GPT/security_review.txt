VULNERABILITY #1
  CWE-798 (Use of Hard-Coded Credentials)
  Severity: Critical
  File: include/app_config.hpp, line 8
  Description: The JWT signing secret has a hardcoded default value
    "development-secret-change-me". If no configuration file or environment
    variable overrides this value, the application runs with a publicly known
    secret. Any attacker who reads the source code (or guesses the default) can
    forge arbitrary JWT tokens, impersonate any user, and gain full access to
    all API endpoints including artist-only operations.
  Exploitation: An attacker crafts a JWT with {"sub":"<any_id>","role":"ARTIST"}
    signed with the known default secret and sends it in the Authorization
    header to access any protected endpoint.

--------------------------------------------------------------------------------

VULNERABILITY #2
  CWE-942 (Permissive Cross-Domain Policy / CORS Misconfiguration)
  Severity: High
  File: src/main.cpp, lines 122-132
  Description: The server sets "Access-Control-Allow-Origin: *" on all
    responses. This allows any website on the internet to make authenticated
    cross-origin requests to the API. Combined with the fact that the JWT token
    can be passed as a query parameter, a malicious website could make API calls
    using a stolen or leaked token.
  Exploitation: An attacker hosts a malicious webpage that makes fetch()
    requests to the streaming API. If the victim has a token stored in
    localStorage or accessible via JavaScript, the attacker site can interact
    with the API on the victim's behalf.

--------------------------------------------------------------------------------

VULNERABILITY #3
  CWE-319 (Cleartext Transmission of Sensitive Information)
  Severity: High
  File: src/main.cpp, line 443
  Description: The server listens on "0.0.0.0" (all interfaces) using plain
    HTTP with no TLS configuration. All data including JWT tokens, passwords,
    and user credentials are transmitted in cleartext.
  Exploitation: An attacker on the same network can perform passive
    eavesdropping or an active man-in-the-middle attack to capture login
    credentials, JWT tokens, and audio streams.

--------------------------------------------------------------------------------

VULNERABILITY #4
  CWE-522 (Insufficiently Protected Credentials)
  Severity: High
  File: src/main.cpp, lines 43-45
  Description: The authorize_request function accepts JWT tokens via the
    "token" query parameter. Query parameters are logged in web server access
    logs, browser history, HTTP Referer headers, and proxy logs, exposing
    authentication tokens in multiple locations where they can be harvested.
  Exploitation: An attacker with access to server logs, proxy logs, or browser
    history can extract valid JWT tokens from URLs and use them to impersonate
    users.

--------------------------------------------------------------------------------

VULNERABILITY #5
  CWE-400 (Uncontrolled Resource Consumption / No Rate Limiting)
  Severity: Medium
  File: src/main.cpp, lines 187-239
  Description: The /api/auth/login endpoint has no rate limiting or account
    lockout mechanism. An attacker can make unlimited login attempts to brute
    force user passwords.
  Exploitation: An attacker automates login requests with common passwords or
    dictionary attacks against known email addresses to compromise accounts.

--------------------------------------------------------------------------------

VULNERABILITY #6
  CWE-79 (Cross-Site Scripting - Stored XSS)
  Severity: Medium
  File: src/main.cpp, lines 263-265, and include/models.hpp lines 65-82
  Description: User-supplied values for song title, genre, description, and
    artist name are stored in the song repository and returned in JSON API
    responses without any output encoding or sanitization. If any frontend
    renders these values using innerHTML or similar mechanisms, stored XSS is
    possible.
  Exploitation: An artist uploads a song with a title containing a JavaScript
    payload (e.g., "<script>alert(1)</script>"). When another user's browser
    renders this value from the API response, the script executes in their
    browser context.

--------------------------------------------------------------------------------

VULNERABILITY #7
  CWE-434 (Unrestricted Upload of File with Dangerous Type)
  Severity: Medium
  File: src/main.cpp, lines 280-284
  Description: File upload validation only checks that the sanitized filename
    ends with ".mp3". There is no validation of the actual file content (magic
    bytes / MIME type detection). An attacker can upload any file type by simply
    naming it with a .mp3 extension.
  Exploitation: An attacker uploads a malicious file with a .mp3 extension. If
    the uploads directory is ever served directly or if another application
    processes the file, malicious content could be executed.
