VULNERABILITY #1
  CWE-798 (Use of Hard-Coded Credentials)
  Severity: Critical
  File: include/app_config.hpp, line 8
  Description: The JWT signing secret has a hardcoded default value
    "development-secret-change-me". If no configuration file or environment
    variable overrides this value, the application runs with a publicly known
    secret. Any attacker who reads the source code (or guesses the default) can
    forge arbitrary JWT tokens, impersonate any user, and gain full access to
    all API endpoints including artist-only operations.
  Exploitation: An attacker crafts a JWT with {"sub":"<any_id>","role":"ARTIST"}
    signed with the known default secret and sends it in the Authorization
    header to access any protected endpoint.

VULNERABILITY #4
  CWE-942 (Permissive Cross-Domain Policy / CORS Misconfiguration)
  Severity: High
  File: src/main.cpp, lines 122-132
  Description: The server sets "Access-Control-Allow-Origin: *" on all
    responses (both the OPTIONS handler on line 125 and the pre-routing handler
    on line 131). This allows any website on the internet to make
    authenticated cross-origin requests to the API. Combined with the fact
    that the JWT token can be passed as a query parameter (line 44), a
    malicious website could make API calls using a stolen or leaked token.
  Exploitation: An attacker hosts a malicious webpage that makes fetch()
    requests to the streaming API. If the victim has a token stored in
    localStorage or accessible via JavaScript, the attacker site can interact
    with the API on the victim's behalf.

VULNERABILITY #7
  CWE-352 (Cross-Site Request Forgery)
  Severity: High
  File: src/main.cpp, lines 138-239, 257-318
  Description: The application has no CSRF protection mechanism. State-changing
    POST endpoints (/api/auth/register, /api/auth/login, /api/artist/songs)
    rely solely on the JWT token in the Authorization header. With the
    permissive CORS policy (Access-Control-Allow-Origin: *), cross-origin
    requests with credentials can be made from any domain. There are no CSRF
    tokens, no SameSite cookie policies, and no Origin/Referer validation.
  Exploitation: A malicious website can craft POST requests to the API
    endpoints. If the victim's browser has cached credentials or the token is
    accessible, actions can be performed without the user's consent.

VULNERABILITY #8
  CWE-347 (Improper Verification of Cryptographic Signature)
  Severity: Medium
  File: src/jwt_service.cpp, lines 50-58
  Description: The JWT verification does not check the "alg" field in the
    header. While the implementation always uses HMAC-SHA256 for verification,
    the header's algorithm claim is completely ignored. Although this custom
    implementation is not vulnerable to the classic "alg:none" attack (since
    it always computes HMAC), the lack of algorithm validation means the code
    does not verify that the token was created with the expected algorithm.
    This is a defense-in-depth issue and a deviation from JWT best practices
    (RFC 7515).
  Exploitation: Limited direct exploitation in the current implementation,
    but if the verification logic is ever extended to support multiple
    algorithms, this omission could become critical.

VULNERABILITY #16
  CWE-330 (Use of Insufficiently Random Values)
  Severity: Medium
  File: src/utils.cpp, lines 207-220
  Description: The generate_id function uses random_bytes(12) to produce
    24-character IDs, but the mapping algorithm (byte % 36 and byte / 36 % 36)
    introduces modulo bias. Since 256 is not divisible by 36, certain
    characters in the alphabet are slightly more likely than others. This
    reduces the effective entropy of generated IDs below the theoretical
    maximum. While this does not make IDs easily guessable, it is a
    cryptographic weakness.
  Exploitation: With a large number of IDs, statistical analysis could reveal
    patterns that narrow the search space for brute-forcing valid song or user
    IDs.

--------------------------------------------------------------------------------

VULNERABILITY #17
  CWE-252 (Unchecked Return Value)
  Severity: Medium
  File: src/utils.cpp, lines 248-268 (sha256_bytes), lines 276-307 (hmac_sha256)
  Description: The BCrypt API calls (BCryptOpenAlgorithmProvider,
    BCryptGetProperty, BCryptCreateHash, BCryptHashData, BCryptFinishHash) do
    not check return values for errors. If any of these calls fail, the
    function continues with uninitialized or zeroed data, potentially producing
    incorrect hashes. This could undermine signature verification or password
    hashing, leading to authentication bypasses.
  Exploitation: Under resource exhaustion or unusual system conditions, a
    BCrypt call fails silently. The resulting hash is incorrect, and signature
    verification may produce unpredictable results, potentially accepting
    invalid tokens.

VULNERABILITY #18
  CWE-613 (Insufficient Session Expiration)
  Severity: Medium
  File: src/jwt_service.cpp, line 12
  Description: JWT tokens have a default expiration of 180 minutes (3 hours).
    There is no token revocation mechanism, no refresh token rotation, and no
    way to invalidate a compromised token before it expires. Once issued, a
    token is valid for the full 3-hour window regardless of password changes
    or account status changes.
  Exploitation: If a token is stolen, the attacker has a 3-hour window of
    unrestricted access. Even if the user changes their password, the old
    token remains valid.

  Severity: Low
  File: src/utils.cpp, line 210
  Description: The generate_id function uses only 12 random bytes to produce

  Description: The BCryptGenRandom call does not check its return status

Critical issues requiring immediate attention:
  1. Hardcoded JWT secret in app_config.hpp (CWE-798)
  2. Hardcoded Firebase service account path with sensitive info (CWE-798)

High-priority issues:
  3. Firebase project ID exposed in source code (CWE-200)
  4. Wildcard CORS policy (CWE-942)
  5. No TLS / plaintext HTTP on all interfaces (CWE-319)
  6. JWT token accepted via query parameter (CWE-522)
  7. No CSRF protection (CWE-352)

Recommendations:
  - Require JWT secret to be provided via environment variable; refuse to start
    with the default value.
  - Remove all hardcoded paths and credentials from source code.
  - Restrict CORS to specific allowed origins.
  - Implement TLS or deploy behind a reverse proxy with TLS termination.
  - Remove query parameter token acceptance; use only Authorization headers.
  - Add rate limiting to authentication endpoints.
  - Implement request body size limits.
  - Add file content validation (magic bytes) for uploads.
  - Sanitize all error messages returned to clients.
  - Check return values of all BCrypt/NCrypt API calls.
  - Implement token revocation or shorter token lifetimes with refresh tokens.
  - Enforce stronger password policies.
