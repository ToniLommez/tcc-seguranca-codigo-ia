VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID: CWE-798
Name: JWT Secret Key Without Fallback Validation (Potential None Secret)
Severity: Critical
File: config.py, line 6 / auth.py, line 17
Description:
  JWT_SECRET_KEY is loaded from an environment variable via os.getenv("JWT_SECRET_KEY").
  If the environment variable is not set, os.getenv returns None. There is no validation
  anywhere in the codebase that JWT_SECRET_KEY is actually set or non-empty. In some
  PyJWT versions, passing None as the key for HS256 may allow trivially forgeable tokens.
Exploitation:
  An attacker who discovers or guesses that the secret key is None/empty can forge
  arbitrary JWT tokens, impersonate any user (including artists), and gain full access
  to the application. They can craft tokens with any user_id and user_type claim.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID: CWE-942
Name: Overly Permissive CORS Configuration
Severity: High
File: app.py, line 11
Description:
  CORS(app) is called with no arguments, which defaults to allowing all origins
  (Access-Control-Allow-Origin: *). This permits any website to make cross-origin
  requests to the API, including reading responses.
Exploitation:
  Any malicious website can make authenticated requests to the API if it obtains a
  user's JWT token (e.g., via XSS on any domain). The permissive CORS policy allows
  the attacker's site to read API responses, enabling data exfiltration.

VULNERABILITY #3

CWE ID: CWE-215
Name: Debug Mode Enabled in Production
Severity: High
File: app.py, line 26
Description:
  The application runs with debug=True (app.run(debug=True, port=5000)). Flask's
  debug mode enables the interactive Werkzeug debugger, which provides a Python

Exploitation:

  code execution on the server, including reading environment variables (such as
  JWT_SECRET_KEY and Firebase credentials) and executing system commands.

CWE ID: CWE-200
Name: Token Exposure in URL Query Parameter
Severity: High
File: routes_songs.py, lines 59-63
Description:
  The stream_song endpoint accepts JWT tokens via the URL query parameter
  (?token=...). Tokens passed in URLs are logged in web server access logs,
  browser history, HTTP Referer headers, and potentially proxy/CDN logs.
Exploitation:
  An attacker with access to server logs, proxy logs, browser history, or network
  monitoring can extract valid JWT tokens from URLs. These tokens can then be used
