VULNERABILITY #1
----------------
CWE-798 - Use of Hard-coded Credentials
Severity: Critical
File: backend/config.py, Line 10
Description: The JWT secret key has a hardcoded default fallback value
"lead-manager-secret-key-change-in-production". If the environment variable
JWT_SECRET_KEY is not set, this weak, predictable secret is used to sign and
verify all authentication tokens.
Exploitation: An attacker who discovers or guesses this default secret can forge
arbitrary JWT tokens, impersonate any user, and gain full access to all accounts
and lead data without authentication.

VULNERABILITY #2
----------------
CWE-942 - Overly Permissive Cross-Origin Resource Sharing (CORS)
Severity: Medium
File: backend/main.py, Lines 8-14
Description: The CORS configuration uses allow_methods=["*"] and
allow_headers=["*"] with allow_credentials=True. While the origins are
restricted to localhost in development, using wildcard methods and headers is
unnecessarily permissive, and any loosening of the origin policy in production
becomes a significant vulnerability.
Exploitation: If the application is deployed with a broader origin policy (a
common misconfiguration), any website could make authenticated cross-origin
requests, stealing data or performing actions on behalf of authenticated users.

VULNERABILITY #3
----------------
CWE-521 - Weak Password Requirements
Severity: Medium
File: backend/models.py, Lines 22-25 and backend/routes_auth.py, Lines 10-29
Description: The UserCreate model and registration endpoint do not enforce any
password complexity, minimum length, or strength requirements on the server side.
The password field is defined as "str" with no validators. The frontend enforces
only minLength={6}, a client-side check that can be trivially bypassed.
Exploitation: An attacker can register accounts with extremely weak passwords
(e.g., "a", "1") via API calls, and weak passwords across the user base are
easier to brute-force.

VULNERABILITY #4
----------------
CWE-770 - Allocation of Resources Without Limits or Throttling (No Rate Limiting)
Severity: Medium
File: backend/routes_auth.py, Lines 32-50 (login) and Lines 9-29 (register)
Description: The authentication endpoints do not implement any rate limiting or
account lockout mechanism. There is no throttling on failed login attempts.
Exploitation: An attacker can perform unlimited brute-force attacks against the
login endpoint to guess user passwords, and abuse the registration endpoint to
create unlimited accounts.

VULNERABILITY #5
----------------
CWE-116 - Improper Encoding or Escaping of Output (CSV Injection)
Severity: Medium
File: backend/routes_leads.py, Lines 236-263
Description: The CSV export endpoint writes lead data directly into CSV format
without sanitizing cell values. Fields like full_name, company, and position
accept arbitrary user input that is later exported to CSV. Values beginning with
=, +, -, or @ can be interpreted as formulas by spreadsheet applications.
Exploitation: An attacker can create a lead with a name like "=CMD('calc')".
When another user exports and opens the CSV file in Excel or Google Sheets, the
malicious formula executes, potentially running arbitrary commands on the
victim's machine (Formula Injection).

VULNERABILITY #6
----------------
CWE-502 - Deserialization of Untrusted Data (via Excel/pandas parsing)
Severity: Medium
File: backend/routes_leads.py, Lines 189-191
Description: The import endpoint uses pandas.read_csv() and pandas.read_excel()
to parse user-uploaded files. The pd.read_excel() function with the openpyxl
engine processes complex XML-based Office documents that can contain embedded
objects and external references.
Exploitation: Crafted Excel files can exploit vulnerabilities in the parsing
libraries. Malicious .xlsx files could trigger XML External Entity (XXE) attacks
during parsing, potentially allowing an attacker to read arbitrary files from
the server or perform Server-Side Request Forgery. Zip bombs disguised as .xlsx
files can consume excessive memory and CPU.

VULNERABILITY #7
----------------
CWE-285 - Improper Authorization (No Role-Based Access Control)
Severity: Low
File: backend/routes_leads.py (all routes) and backend/auth.py, Lines 26-43
Description: The application has no role-based access control (RBAC) system. All
authenticated users have identical permissions. While leads are segregated by
user ID, there is no admin role, no concept of shared leads, and no granular
permissions.
Exploitation: This is primarily a design limitation. In a multi-user business
context, there is no way to grant read-only access, limit deletion capabilities,
or create supervisory roles.
