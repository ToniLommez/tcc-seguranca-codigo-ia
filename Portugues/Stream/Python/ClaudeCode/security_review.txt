VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID: CWE-798 (Use of Hard-Coded Credentials)
Vulnerability: Hardcoded Secret Key
Severity: Critical
File: backend/config.py
Line: 7
Description:
  The JWT secret key has a hardcoded default value:
  SECRET_KEY = os.getenv("SECRET_KEY", "soundstream-secret-key-2024-mude-em-producao")
  If the SECRET_KEY environment variable is not set, the application falls back
  to a publicly visible, predictable secret key embedded in source code.
Exploitation:
  An attacker who reads the source code can use this default key to forge
  arbitrary JWT tokens, impersonate any user (including artists), and gain full
  unauthorized access to all application functionality including uploading
  malicious files.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID: CWE-942 (Overly Permissive Cross-Origin Resource Sharing)
Vulnerability: Unrestricted CORS Policy
Severity: High
File: backend/main.py
Lines: 14-20
Description:
  The CORS middleware is configured to allow all origins, all methods, and
  all headers with credentials enabled (allow_origins=["*"],
  allow_credentials=True). This combination allows any website on the internet
  to make authenticated cross-origin requests to the API.
Exploitation:
  An attacker can host a malicious website that makes cross-origin requests to
  the SoundStream API. If a logged-in user visits the malicious site, the
  attacker's JavaScript can invoke API endpoints on behalf of the victim,
  including uploading songs, streaming content, and accessing user data.

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE ID: CWE-319 (Cleartext Transmission of Sensitive Information)
Vulnerability: JWT Token Exposed in URL Query Parameter
Severity: High
File: backend/music/router.py
Lines: 89, 92-98
Description:
  The stream endpoint accepts JWT authentication tokens via URL query
  parameter: token: Optional[str] = Query(default=None). JWT tokens contain
  user identity and role information.
Exploitation:
  Tokens passed in URL query strings are logged in web server access logs,
  browser history, proxy logs, CDN logs, and HTTP Referer headers. An attacker
  with access to any of these log sources can steal valid JWT tokens and
  impersonate users for the 24-hour validity window.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE ID: CWE-918 (SSRF) / CWE-22 (Path Traversal)
Vulnerability: File Path Stored in Database Used Directly for File Access
Severity: Medium
File: backend/music/router.py
Lines: 110-116
Description:
  The streaming endpoint retrieves the file_path from the database and uses it
  directly to open and serve the file (with open(file_path, "rb") as f). The
  file_path is an absolute filesystem path stored in the database. If an
  attacker can manipulate the database record, they could point file_path to
  any file on the server.
Exploitation:
  If the database is compromised or a future feature allows modifying song
  metadata, an attacker could change file_path to point to sensitive files like
  application configuration files or the Firebase credentials JSON file, and
  then stream those files through the API.

VULNERABILITY #5
--------------------------------------------------------------------------------
CWE ID: CWE-521 (Weak Password Requirements)
Vulnerability: Insufficient Password Complexity Requirements
Severity: Medium
File: backend/auth/router.py
Lines: 55-56
Description:
  The only password validation is a minimum length of 6 characters
  (if len(req.password) < 6). There are no requirements for uppercase letters,
  lowercase letters, digits, special characters, or checks against common
  passwords.
Exploitation:
  Users can set extremely weak passwords like "123456" or "aaaaaa", which are
  trivially guessable. Combined with the lack of rate limiting, this makes
  brute-force attacks highly effective.

Top priority remediations:

2. Restrict CORS origins to specific trusted domains instead of "*".
3. Stop accepting JWT tokens in URL query parameters; use the Authorization
   header or HttpOnly cookies for the streaming endpoint.
4. Validate that the stored file_path resides within the uploads directory.
5. Strengthen password requirements.
