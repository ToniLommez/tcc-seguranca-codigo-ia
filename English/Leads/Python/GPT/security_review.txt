VULNERABILITY #1
================================================================================
CWE ID:       CWE-1078 / CWE-16
Name:         Overly Permissive CORS Configuration
Severity:     High
File:         app/main.py
Line(s):      16-22
Description:
  The CORS middleware is configured with allow_origins=["*"],
  allow_credentials=True, allow_methods=["*"], and allow_headers=["*"].
  This allows any origin on the internet to make authenticated,
  credentialed cross-origin requests to the API. The combination of
  wildcard origins with allow_credentials=True is especially dangerous
  because it means a malicious website can make requests on behalf of a
  logged-in user and read the responses.
Exploitation:
  An attacker hosts a malicious webpage that makes fetch() requests to
  the Lead Management API. If a user visits the attacker's page while
  authenticated, the attacker's JavaScript can read, modify, and delete
  all of that user's leads, exfiltrate customer data, or perform any
  action the authenticated user can.

VULNERABILITY #2
================================================================================
CWE ID:       CWE-798
Name:         Hardcoded Default JWT Secret Key
Severity:     Critical
File:         app/core/config.py
Line(s):      9
Description:
  The JWT secret key has a hardcoded default value of
  "change-me-before-production". If the application is deployed without
  setting the JWT_SECRET_KEY environment variable, all tokens will be
  signed with this publicly known secret. Any attacker who knows this
  default value can forge arbitrary JWT tokens, impersonating any user.
Exploitation:
  An attacker crafts a JWT token with {"sub": "<target_user_id>",
  "email": "victim@example.com", "exp": <future_timestamp>}, signs it
  with the key "change-me-before-production", and sends it as a Bearer
  token. The server accepts the forged token and grants the attacker
  full access to the victim's account.

VULNERABILITY #8
================================================================================
CWE ID:       CWE-400
Name:         Unrestricted File Upload Size on Import
Severity:     High
File:         app/api/leads.py, app/services/lead_service.py
Line(s):      leads.py:64-66, lead_service.py:323-373
Description:
  The /api/leads/import endpoint accepts file uploads with no
  restriction on file size. The import_leads function reads the entire
  uploaded file into memory using pd.read_csv() or pd.read_excel()
  without any size limits. There is no validation on the number of rows
  or the file size before processing begins.
Exploitation:
  An attacker uploads an extremely large CSV or XLSX file (hundreds of
  MB or more), causing excessive memory consumption and potentially
  crashing the server (Denial of Service). Even a moderately large file
  with millions of rows would cause the server to become unresponsive
  while iterating and writing each row to Firestore.

VULNERABILITY #9
================================================================================
CWE ID:       CWE-918 / CWE-611
Name:         Potential XML External Entity (XXE) via Excel File Parsing
Severity:     Medium
File:         app/services/lead_service.py
Line(s):      327-328
Description:
  The import_leads function uses pd.read_excel() to parse uploaded
  XLSX/XLS files. XLSX files are ZIP archives containing XML documents.
  Depending on the version of the underlying XML parser (openpyxl or
  xlrd), this could be vulnerable to XXE injection if the parser does
  not disable external entity resolution by default.
Exploitation:
  An attacker crafts a malicious XLSX file containing an XXE payload in
  its internal XML structure. When parsed by the server, it could
  trigger Server-Side Request Forgery (SSRF), read local files on the
  server, or cause denial of service via entity expansion (billion
  laughs attack).

Severity:     High
File:         app/api/auth.py
Line(s):      13-30
Description:
  The /api/auth/login and /api/auth/register endpoints have no rate
  limiting. An attacker can make unlimited login attempts to brute-force
  user passwords, or create unlimited accounts via the register
  endpoint. There is no account lockout mechanism after failed login
  attempts.
Exploitation:

  passwords per second. Similarly, the register endpoint can be abused
  to flood the database with fake user accounts.

VULNERABILITY #13
================================================================================
CWE ID:       CWE-942
Name:         Missing Security Headers
Severity:     High
File:         app/main.py
Line(s):      14 (entire application)
Description:
  The application does not set any security-related HTTP headers. The
  following headers are absent:
    - Strict-Transport-Security (HSTS)
    - X-Content-Type-Options
    - X-Frame-Options / Content-Security-Policy frame-ancestors
    - Content-Security-Policy
    - X-XSS-Protection
    - Referrer-Policy
    - Permissions-Policy
  Without these headers, the application is more susceptible to
  clickjacking, MIME-type sniffing attacks, and other client-side
  attacks. The Jinja2 templates served on frontend routes are
  especially at risk without Content-Security-Policy.
Exploitation:
  An attacker can embed the application in an iframe on a malicious
  site (clickjacking). Without Content-Security-Policy, inline scripts
  injected via any XSS vector execute without restriction. Without
  HSTS, users may connect over HTTP and be subject to man-in-the-middle
  attacks.
