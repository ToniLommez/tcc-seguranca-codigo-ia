VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID: CWE-798 / CWE-1393
Name: JWT Secret Key Without Fallback Validation (Potential None Secret)
Severity: Critical
File: config.py, line 6 / auth.py, line 17
Description:
  JWT_SECRET_KEY is loaded from an environment variable via os.getenv("JWT_SECRET_KEY").
  If the environment variable is not set, os.getenv returns None. There is no validation
  anywhere in the codebase that JWT_SECRET_KEY is actually set or non-empty. When
  JWT_SECRET_KEY is None, the PyJWT library's behavior with a None key can lead to
  tokens being signed with an empty or predictable value, or may cause runtime errors
  that leak information. In some PyJWT versions, passing None as the key for HS256 may
  allow trivially forgeable tokens.
Exploitation:
  An attacker who discovers or guesses that the secret key is None/empty can forge
  arbitrary JWT tokens, impersonate any user (including artists), and gain full access
  to the application. They can craft tokens with any user_id and user_type claim.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID: CWE-352
Name: Cross-Site Request Forgery (CSRF) - No CSRF Protection
Severity: High
File: app.py, line 11; routes_auth.py (all POST endpoints)
Description:
  The application uses CORS(app) with no origin restrictions (allows all origins by
  default) and has no CSRF protection mechanism. While the API uses JWT Bearer tokens
  for most endpoints, the registration and login endpoints at /api/auth/register and
  /api/auth/login have no CSRF protection and accept POST requests from any origin.
Exploitation:
  A malicious website could craft requests to the registration or login endpoints.
  Combined with the permissive CORS policy, an attacker could potentially perform
  login CSRF attacks to associate a victim's session with an attacker-controlled
  account, or perform mass account registration.

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE ID: CWE-942
Name: Overly Permissive CORS Configuration
Severity: High
File: app.py, line 11
Description:
  CORS(app) is called with no arguments, which defaults to allowing all origins
  (Access-Control-Allow-Origin: *). This permits any website to make cross-origin
  requests to the API, including reading responses.
Exploitation:
  Any malicious website can make authenticated requests to the API if it obtains a
  user's JWT token (e.g., via XSS on any domain). The permissive CORS policy allows
  the attacker's site to read the API responses, enabling data exfiltration of song
  listings, user information, and other sensitive data.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE ID: CWE-200
Name: Token Exposure in URL Query Parameter
Severity: High
File: routes_songs.py, lines 59-63
Description:
  The stream_song endpoint accepts JWT tokens via the URL query parameter
  (?token=...). Tokens passed in URLs are logged in web server access logs,
  browser history, HTTP Referer headers, and potentially proxy/CDN logs. This
  significantly increases the attack surface for token theft.
Exploitation:
  An attacker with access to server logs, proxy logs, browser history, or network
  monitoring can extract valid JWT tokens from URLs. These tokens can then be used
  to impersonate users and access the full API for the token's validity period (24h).

VULNERABILITY #5
--------------------------------------------------------------------------------
CWE ID: CWE-215
Name: Debug Mode Enabled in Production
Severity: High
File: app.py, line 26
Description:
  The application runs with debug=True (app.run(debug=True, port=5000)). Flask's
  debug mode enables the interactive Werkzeug debugger, which provides a Python
  console in the browser when an exception occurs. This also enables automatic code
  reloading and exposes detailed stack traces.
Exploitation:
  If an attacker can trigger an exception (e.g., by sending malformed input), they
  gain access to an interactive Python shell in the browser. This allows arbitrary
  code execution on the server, including reading files, environment variables
  (such as JWT_SECRET_KEY and Firebase credentials), and executing system commands.
  Even without the PIN, detailed stack traces leak source code, file paths, and
  internal application structure.

VULNERABILITY #8
--------------------------------------------------------------------------------
CWE ID: CWE-285
Name: Authorization Bypass via JWT Claim Manipulation (Role in Token)
Severity: Medium
File: auth.py, lines 31 and 42; routes_auth.py, lines 89-95
Description:
  The user's role (user_type) is embedded directly in the JWT token at login time
  (line 93 of routes_auth.py) and trusted without re-verification against the
  database on subsequent requests (auth.py lines 31, 42). If a user's role is
  changed in the database (e.g., an artist account is downgraded), the JWT token
  remains valid with the old role for up to 24 hours. There is no token revocation
  mechanism.
Exploitation:
  A user whose privileges are revoked retains full access until their token expires.
  If the JWT_SECRET_KEY is compromised (see Vulnerability #1), an attacker can
  craft tokens with arbitrary roles. Additionally, there is no admin role or
  mechanism to invalidate compromised tokens.

VULNERABILITY #10
--------------------------------------------------------------------------------
CWE ID: CWE-400
Name: Denial of Service via Unvalidated Range Header
Severity: Medium
File: routes_songs.py, lines 85-92
Description:
  The Range header parsing in the stream_song endpoint does not validate that
  byte_start is less than byte_end, that byte_end is less than file_size, or that
  byte_start is non-negative. Malformed Range headers (e.g., "bytes=999999999-0"
  or "bytes=-1-") can cause unexpected behavior, integer overflow, or excessive
  memory consumption.
Exploitation:
  An attacker can send crafted Range headers to cause the server to attempt reading
  invalid byte ranges, potentially causing crashes, excessive memory use, or
  information leakage through error messages. A Range header with a very large
  byte_start could cause the server to seek past the end of the file.

VULNERABILITY #13
--------------------------------------------------------------------------------
CWE ID: CWE-331
Name: Insufficient Password Hashing Iterations
Severity: Low
File: routes_auth.py, lines 12-16
Description:
  The password hashing uses PBKDF2-HMAC-SHA256 with 100,000 iterations. While
  this was acceptable in earlier years, current OWASP recommendations suggest
  at least 600,000 iterations for PBKDF2-SHA256 (as of 2023). More importantly,
  PBKDF2 is less resistant to GPU-based attacks compared to modern alternatives
  like bcrypt, scrypt, or Argon2.
Exploitation:
  If the database is compromised, an attacker with GPU resources can crack
  passwords faster than with a properly configured modern hashing algorithm.
  The 100,000 iteration count is below current best-practice thresholds.

VULNERABILITY #14
--------------------------------------------------------------------------------
CWE ID: CWE-918
Name: Potential SSRF via Unvalidated Firebase Document ID
Severity: Medium
File: routes_artist.py, line 75; routes_songs.py, line 72
Description:
  The song_id parameter in delete_song and stream_song endpoints is taken directly
  from the URL path and used in Firestore document lookups without sanitization.
  While Firestore itself is not directly vulnerable to injection, specially crafted
  document IDs containing path separators (e.g., "../other_collection/doc") could
  potentially access unintended Firestore document paths depending on the Firestore
  SDK's handling of path components.
Exploitation:
  An attacker could attempt to craft song_id values with path traversal characters
  to access or reference documents outside the intended "songs" collection. The
  impact depends on Firestore SDK behavior with path-like characters in document IDs.
