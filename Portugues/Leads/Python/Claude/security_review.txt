VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID:        CWE-798 (Use of Hard-Coded Credentials)
Name:          Hardcoded JWT Secret Key
Severity:      CRITICAL
File:          backend/config.py
Line:          4
Code:          SECRET_KEY = os.getenv("SECRET_KEY", "claude-python-leads-secret-2024-change-in-production")

Description:
The JWT signing secret key has a hardcoded default fallback value
"claude-python-leads-secret-2024-change-in-production". If the SECRET_KEY
environment variable is not set (which is common in development, staging, or
misconfigured production environments), the application will use this
predictable, publicly visible secret to sign and verify all authentication
tokens.

Exploitation:
An attacker who reads this source code (e.g., from a public repository or
leaked source) can forge valid JWT tokens for any user. This allows complete
authentication bypass -- the attacker can impersonate any user, including
administrators, without knowing their credentials. The attacker simply
creates a JWT payload with the desired user identity, signs it with the
known secret, and sends it in API requests.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID:        CWE-942 (Overly Permissive Cross-domain Whitelist)
Name:          Unrestricted CORS Policy (allow_origins=["*"])
Severity:      HIGH
File:          backend/main.py
Lines:         14-20
Code:
    app.add_middleware(
        CORSMiddleware,
        allow_origins=["*"],
        allow_credentials=True,
        allow_methods=["*"],
        allow_headers=["*"],
    )

Description:
The CORS middleware is configured to allow requests from ANY origin ("*"),
with ALL methods and ALL headers permitted, and credentials are allowed.
This is the most permissive CORS configuration possible. While browsers
may block the combination of allow_origins=["*"] with
allow_credentials=True, many CORS implementations reflect the Origin
header when "*" is used, effectively allowing credentialed cross-origin
requests from any domain.

Exploitation:
An attacker can host a malicious website that makes cross-origin requests
to this API using the victim's credentials (cookies or tokens). If a
logged-in user visits the attacker's page, JavaScript on that page can
make authenticated API requests to the Lead Management system, reading
sensitive lead data, creating or modifying leads, or performing any action
the victim is authorized to do. This is a classic cross-site request
forgery / cross-origin data theft vector.

2. The service account identifier ("fbsvc-375221693a")
3. The exact filesystem location of the credentials file (five

account JSON file. The exposed project ID and service account ID
can be used to craft targeted attacks against the Firebase project.
If the credentials file is accidentally committed to version control
or exposed through a directory listing, the attacker gains full
Firebase Admin access.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE ID:        CWE-22 (Improper Limitation of a Pathname to a Restricted Directory)
Name:          Potential Path Traversal in Static File Serving
Severity:      MEDIUM
File:          backend/main.py
Lines:         30-32
Code:
    def _html(name: str):
        path = FRONTEND_DIR / name
        return FileResponse(str(path), media_type="text/html")

Description:
The _html() helper function takes a filename and constructs a file path
by joining it with FRONTEND_DIR. While this function is currently called
only with hardcoded filenames (e.g., "index.html", "dashboard.html"),
the function itself does not validate or sanitize the "name" parameter.
If this function is ever called with user-controlled input in a future
code change, it would be vulnerable to path traversal attacks. This
represents a latent vulnerability / insecure design pattern.

Exploitation:
If _html() were called with user input such as "../../etc/passwd" or
"../backend/config.py", an attacker could read arbitrary files from the
server filesystem. While the current code only calls _html() with
hardcoded strings, the function's lack of input validation makes it a
ticking time bomb for future developers who might use it with dynamic
input.

VULNERABILITY #5

CWE ID:        CWE-331 (Insufficient Entropy)
Name:          Weak JWT Secret with Insufficient Entropy
Severity:      MEDIUM
File:          backend/config.py
Line:          4
Code:          SECRET_KEY = os.getenv("SECRET_KEY", "claude-python-leads-secret-2024-change-in-production")

Description:
Even if the hardcoded default is overridden by an environment variable,
there is no validation that the provided secret has sufficient entropy
or minimum length. The application accepts any string as the JWT signing
secret, including weak or short values. The hardcoded default itself is
a human-readable string with low entropy relative to its length,

Exploitation:

"123456"), an attacker could brute-force the JWT signing key using tools
like jwt-cracker or hashcat, then forge arbitrary authentication tokens.

VULNERABILITY #6
--------------------------------------------------------------------------------
CWE ID:        CWE-614 (Sensitive Cookie in HTTPS Session Without 'Secure' Attribute)
               CWE-1004 (Sensitive Cookie Without 'HttpOnly' Flag)
Name:          No Secure Cookie/Token Transport Configuration
Severity:      MEDIUM
File:          backend/config.py, backend/main.py
Lines:         config.py:5-6, main.py:57

Description:
The application configures JWT tokens (HS256 algorithm, 24-hour
expiration) but there is no configuration for secure token transport.
There are no settings for Secure, HttpOnly, or SameSite cookie
attributes. The server binds to 0.0.0.0:8000 without TLS. The 24-hour
token lifetime (ACCESS_TOKEN_EXPIRE_MINUTES = 1440) is excessively long
for an access token, increasing the window of exploitation if a token is
stolen.

Exploitation:
Tokens transmitted over unencrypted HTTP connections can be intercepted
by network attackers (man-in-the-middle). The long 24-hour lifetime means
a stolen token remains valid for an extended period. If tokens are stored
in JavaScript-accessible storage (localStorage), they are vulnerable to
XSS-based theft.

Name:          No Authorization Middleware or Access Control Framework
Severity:      HIGH
File:          backend/main.py
Lines:         12-23

Description:
The main application file sets up routers for authentication and leads
but does not configure any authorization middleware, role-based access
control, or resource ownership verification at the application level.
While individual route handlers (in the out-of-scope routes.py files) may
implement access controls, the absence of any centralized authorization
framework means each endpoint must independently implement access
checks, which is error-prone.

Note: The auth and leads route files (auth/routes.py, leads/routes.py)
were not included in the review scope but are critical for a complete
assessment of authorization controls.

Exploitation:
If any route handler fails to implement its own authorization check, the
endpoint becomes accessible to any authenticated user (or even
unauthenticated users). Without centralized authorization, there is no
safety net -- a single developer oversight exposes the endpoint. Common
consequences include IDOR (Insecure Direct Object Reference) where one
user can access or modify another user's leads.
