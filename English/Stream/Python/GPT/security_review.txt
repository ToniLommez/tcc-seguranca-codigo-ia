--- VULNERABILITY 1 ---
CWE ID: CWE-798 (Use of Hard-coded Credentials)
Name: Hardcoded JWT Secret Key
Severity: Critical
File: backend/app/core/settings.py
Line: 9
Description:
  The JWT secret key is hardcoded as a default value:
  jwt_secret_key: str = "change-this-secret-key"
  While pydantic_settings can override this from environment variables or a
  .env file, the default value is a weak, guessable string. If no .env file
  is provided or the environment variable is not set, the application runs
  with this trivially guessable secret.
Exploitation:
  An attacker who discovers (or guesses) the default secret can forge
  arbitrary JWT tokens, impersonate any user (including artists), upload
  malicious files, and access all protected endpoints. The default value
  "change-this-secret-key" is easily discoverable in the source code.

--- VULNERABILITY 4 ---
CWE ID: CWE-319 (Cleartext Transmission of Sensitive Information)
Name: JWT Token Accepted via Query Parameter
Severity: High
File: backend/app/api/dependencies.py
Lines: 17-21
Description:
  The get_current_user function accepts the JWT access token via a query
  parameter (access_token: str | None = Query(default=None)) as a fallback
  when no Authorization header is provided. This means authentication
  tokens can be passed in the URL.
Exploitation:
  JWT tokens passed as query parameters are logged in web server access
  logs, browser history, proxy logs, Referer headers, and any URL logging
  middleware. An attacker who gains access to any of these log sources
  obtains valid session tokens and can impersonate users. This also makes
  tokens more susceptible to shoulder surfing and URL sharing accidents.

--- VULNERABILITY 5 ---
CWE ID: CWE-89 (SQL Injection)
Name: Potential SQL Injection via ILIKE with User Input
Severity: Medium
File: backend/app/services/song_service.py
Lines: 51-56
Description:
  The search_songs function constructs ILIKE filter clauses using
  f-string interpolation of user-supplied values directly into the
  SQL pattern:
    Song.title.ilike(f"%{title.strip()}%")
    Song.artist_name.ilike(f"%{artist.strip()}%")
    Song.genre.ilike(f"%{genre.strip()}%")
  While SQLAlchemy's ORM does parameterize queries, the LIKE/ILIKE
  wildcards (% and _) within the user input are NOT escaped. This allows
  LIKE-injection where users can craft patterns with special wildcard
  characters to manipulate query logic.
Exploitation:
  An attacker can submit search terms containing "%" or "_" characters to
  craft wildcard patterns that bypass intended search behavior, potentially
  enumerating all songs or causing performance degradation through
  expensive pattern matching. For example, searching with "%" returns all
  records. While this is not full SQL injection due to SQLAlchemy's
  parameterization, the unescaped LIKE wildcards constitute a form of
  query manipulation (CWE-89 variant).

--- VULNERABILITY 6 ---

Name: No CSRF Protection
Severity: High
File: backend/app/main.py, backend/app/api/auth.py, backend/app/api/artist.py
Lines: N/A (architectural issue)
Description:
  The application has no CSRF protection mechanism. There are no CSRF

  or query parameters), and no origin validation beyond the permissive
  CORS configuration. Combined with the wildcard CORS policy

  the API endpoints on behalf of authenticated users. If a user visits
  the attacker's page while authenticated, the attacker can perform
  actions such as uploading songs or modifying the user's session.

--- VULNERABILITY 7 ---
CWE ID: CWE-434 (Unrestricted Upload of File with Dangerous Type)
Name: Insufficient File Upload Validation
Severity: High
File: backend/app/api/artist.py
Lines: 60-65
Description:
  The song upload endpoint only validates the file extension:
    extension = Path(file.filename or "").suffix.lower()
    if extension != ".mp3":
  This check relies solely on the client-provided filename extension.
  There is no validation of:
  - The file's actual content/MIME type (magic bytes)
  - File size limits
  - The Content-Type header of the uploaded file
  An attacker can upload any file content with a ".mp3" extension.
Exploitation:
  An attacker (authenticated as an artist) can upload a malicious file
  (e.g., executable, PHP shell, HTML with JavaScript) renamed with a .mp3
  extension. Since file_path is stored as an absolute path in the database
  and the uploads directory is accessible, this could lead to stored XSS
  if the file is served with incorrect content type, or server compromise
  if the file is later executed. There is also no file size limit, allowing
  denial-of-service through uploading extremely large files.

Name: Unsanitized File Path from Database Used for File Serving
Severity: High
File: backend/app/api/songs.py
Lines: 106-112
Description:
  The stream_song endpoint reads the file path directly from the database
  record without validating that it resides within the expected uploads
  directory:

    if not file_path.exists():
  The file_path stored in the database is an absolute path (stored in
  artist.py line 82 as str(target_path)). If an attacker manages to
  manipulate the database record (e.g., through SQL injection in another
  part of the application, or direct database access), they can set
  file_path to any file on the server's filesystem.
Exploitation:
  If the database is compromised or a song record's file_path is
  manipulated, an attacker can read arbitrary files from the server's
  filesystem by requesting the stream endpoint for that song. This could
  expose configuration files, source code, credentials, or system files.

--- VULNERABILITY 10 ---
CWE ID: CWE-200 (Exposure of Sensitive Information)
Name: Password Hash Returned in User Data Dictionary
Severity: Medium
File: backend/app/services/user_service.py
Lines: 46-56
Description:
  The create_user function returns the full user_data dictionary which
  includes the "password_hash" field. This dictionary is then passed
  through the application. While the API response schemas (UserResponse,
  TokenResponse) do filter the fields, the password hash flows through
  internal functions and could be inadvertently exposed through error
  messages, logging, or future code changes.
Exploitation:
  If any middleware, error handler, or logging mechanism serializes the
  user dictionary (e.g., during debugging or in an error response), the
  bcrypt password hash could be exposed. While bcrypt is a strong hash,
  exposing hashes is a defense-in-depth violation and could enable offline
  brute-force attacks against weak passwords.

Name: No Token Revocation or Blacklisting Mechanism
Severity: Medium
File: backend/app/core/security.py
Lines: 20-30, backend/app/api/dependencies.py Lines: 17-43
Description:
  The JWT-based authentication system has no mechanism for token
  revocation, logout, or blacklisting. Once a token is issued, it remains
  valid until its expiration time (120 minutes by default). There is no
  logout endpoint and no way to invalidate compromised tokens.
Exploitation:
  If a token is stolen (e.g., via the query parameter exposure described
  in Vulnerability 4, or through log files), the attacker has a guaranteed
  120-minute window of access. The legitimate user cannot revoke the
  stolen token, and there is no mechanism for administrators to force
  session termination.

ADDITIONAL OBSERVATIONS
================================================================================

1. POSITIVE: Password hashing uses bcrypt via passlib (security.py line 9),
   which is a strong and appropriate choice for password storage.

2. POSITIVE: File uploads are saved with UUID-generated filenames
   (artist.py line 67), preventing filename collision and directory
   traversal in the upload path itself.

3. POSITIVE: Pydantic schemas enforce basic input validation with field
   constraints (min_length, max_length on passwords and names).

4. POSITIVE: SQLAlchemy ORM is used for database queries, which provides
   parameterized queries and mitigates most SQL injection vectors.

5. POSITIVE: Role-based access control is implemented via the require_role
   dependency (dependencies.py lines 46-55), separating ARTIST and USER
   permissions.

6. CONCERN: The application uses SQLite (database.py line 7) which is not
   suitable for production multi-user workloads and has limited security
   features compared to PostgreSQL or MySQL.

7. CONCERN: No input length validation on Form fields in the upload
   endpoint (artist.py lines 55-57) for title, genre, and description.
   While the database column has a max length (String(255)), exceeding it
   would cause a database error rather than a clean validation error.
