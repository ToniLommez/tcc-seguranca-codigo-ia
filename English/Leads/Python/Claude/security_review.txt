VULNERABILITY #1
----------------
CWE-798 - Use of Hard-coded Credentials
Severity: Critical
File: backend/config.py, Line 10
Description: The JWT secret key has a hardcoded default fallback value
"lead-manager-secret-key-change-in-production". If the environment variable
JWT_SECRET_KEY is not set, this weak, predictable secret is used to sign and
verify all authentication tokens.
Exploitation: An attacker who discovers or guesses this default secret (which is
committed to source code) can forge arbitrary JWT tokens, impersonate any user,
and gain full access to all accounts and lead data without authentication.

VULNERABILITY #3

Severity: High
File: backend/config.py, Line 10
Description: Even when the default value is used, the secret

VULNERABILITY #4
----------------
CWE-521 - Weak Password Requirements
Severity: Medium
File: backend/models.py, Lines 22-25 and backend/routes_auth.py, Lines 10-29
Description: The UserCreate model and registration endpoint do not enforce any
password complexity, minimum length, or strength requirements on the server side.
The password field is simply defined as "str" with no validators. The frontend
Register.jsx enforces only minLength={6} (line 45) but this is a client-side

Exploitation: An attacker can register accounts with extremely weak passwords

accounts if they also used weak passwords. The server accepts any non-empty
string as a password.

VULNERABILITY #5
----------------
CWE-942 - Overly Permissive Cross-Origin Resource Sharing (CORS)
Severity: Medium
File: backend/main.py, Lines 8-14
Description: The CORS configuration uses allow_methods=["*"] and
allow_headers=["*"], which permits all HTTP methods and all headers from the
specified origins. While the origins are restricted to localhost, using wildcard
methods and headers is unnecessarily permissive. In production, if origins are
changed to be more permissive or if a wildcard origin is added, this becomes a
significant vulnerability.
Exploitation: If the application is deployed with a broader origin policy
(common misconfiguration), any website could make authenticated cross-origin
requests. The allow_credentials=True combined with permissive headers enables
cross-origin attacks to steal data or perform actions on behalf of authenticated
users.

VULNERABILITY #6
----------------
CWE-613 - Insufficient Session Expiration
Severity: Medium
File: backend/config.py, Line 12 and backend/auth.py, Lines 19-23
Description: JWT tokens have a 24-hour expiration time and there is no token
revocation mechanism. Once a token is issued, it remains valid for the full 24
hours regardless of whether the user logs out, changes their password, or their
account is compromised.
Exploitation: If a token is stolen (via XSS, network interception, or device
theft), the attacker has a full 24-hour window to use it. The legitimate user
cannot invalidate the stolen token by logging out, as logout only removes the
token from localStorage on the client side (frontend/src/App.jsx, lines 30-34)
without any server-side invalidation.

VULNERABILITY #7
----------------
CWE-922 - Insecure Storage of Sensitive Information
Severity: Medium
File: frontend/src/App.jsx, Lines 24-27 and frontend/src/services/api.js, Line 10
Description: JWT authentication tokens and user data are stored in the browser's
localStorage. LocalStorage is accessible to any JavaScript running on the same
origin.
Exploitation: If any XSS vulnerability exists in the application (or in any
third-party script loaded on the same origin), an attacker can read the JWT token
from localStorage via document.cookie or localStorage.getItem('token') and use
it to impersonate the victim. HttpOnly cookies would be a more secure storage
mechanism as they are not accessible via JavaScript.

VULNERABILITY #8
----------------
CWE-352 - Cross-Site Request Forgery (CSRF)
Severity: Medium
File: backend/main.py, Lines 8-14 and backend/routes_leads.py (all state-changing routes)
Description: The application does not implement any CSRF protection mechanism.
While the use of Bearer tokens in the Authorization header provides some implicit
CSRF protection (since browsers do not automatically attach custom headers), the
CORS configuration with allow_credentials=True could enable CSRF attacks if the
origin restrictions are loosened. Additionally, there are no CSRF tokens or
SameSite cookie attributes in use.
Exploitation: If CORS origins are misconfigured in production or a subdomain is
compromised, an attacker could craft a malicious page that makes cross-origin
requests to the API. Combined with the Bearer token stored in localStorage, an
XSS-assisted CSRF attack could create, modify, or delete leads.

VULNERABILITY #9
----------------
CWE-400 - Uncontrolled Resource Consumption (Denial of Service via File Upload)
Severity: Medium
File: backend/routes_leads.py, Lines 182-233
Description: The CSV/Excel import endpoint (/import/csv) does not enforce any
file size limit. The file content is read entirely into memory (line 187:
content = file.file.read()), then parsed with pandas. There is no limit on the
number of rows processed.
Exploitation: An attacker can upload an extremely large CSV or Excel file (e.g.,

causing a denial-of-service. Each row also triggers a Firestore write operation,
which could additionally exhaust database quotas and incur costs.

VULNERABILITY #10
-----------------
CWE-434 - Unrestricted Upload of File with Dangerous Type
Severity: Medium
File: backend/routes_leads.py, Lines 183-184
Description: The file type validation only checks the file extension
(file.filename.endswith((".csv", ".xlsx", ".xls"))). It does not validate the
actual file content type (MIME type) or the file's magic bytes. The filename is
controlled by the client and can be easily spoofed.
Exploitation: An attacker can upload a malicious file with a .csv extension that
contains non-CSV content. While the pandas parsing may fail, crafted Excel files
(.xlsx/.xls) can exploit vulnerabilities in the openpyxl or xlrd parsing
libraries (XML External Entity attacks, zip bombs, etc.). Additionally, the
filename is never sanitized and is used directly from client input.

VULNERABILITY #17
-----------------
CWE-116 - Improper Encoding or Escaping of Output (CSV Injection)
Severity: Medium
File: backend/routes_leads.py, Lines 236-263
Description: The CSV export endpoint writes lead data directly into CSV format
without sanitizing cell values. Fields like full_name, company, and position

VULNERABILITY #23

CWE-79 - Cross-Site Scripting (XSS) via dangerouslySetInnerHTML-equivalent patterns
Severity: Low
File: frontend/src/pages/LeadDetail.jsx, Lines 95-104 and frontend/src/pages/Leads.jsx, Lines 147-159
Description: While React escapes content rendered via JSX expressions by default
(mitigating most XSS), the application renders user-controlled data (lead names,

the API returns HTML in error messages or if future code changes use innerHTML,
an attacker could inject malicious scripts through lead data fields that execute
when other users view the lead.

Critical Issues Requiring Immediate Attention:
1. Hardcoded JWT secret key (CWE-798) - Enables complete authentication bypass
2. Exposed Firebase credentials path (CWE-798) - Information disclosure
3. Weak JWT secret with no entropy enforcement (CWE-259)

Key Recommendations:
1. Generate a cryptographically random JWT secret of at least 256 bits and
   require it via environment variable without a fallback default.
2. Remove the hardcoded Firebase credentials path from source code.
3. Implement server-side password strength validation (minimum length,
   complexity requirements).
4. Add rate limiting to authentication endpoints.
5. Implement file size limits and content-type validation on file uploads.
6. Sanitize CSV export output to prevent formula injection.
7. Add HTTPS enforcement for production deployments.
8. Consider using HttpOnly cookies for token storage instead of localStorage.
9. Implement token revocation / server-side session management.
10. Add audit logging for sensitive operations (login, export, delete).
11. Restrict OpenAPI documentation access in production (/docs, /openapi.json).
12. Bind the development server to 127.0.0.1 instead of 0.0.0.0.
