VULNERABILITY #1
================================================================================
CWE ID:       CWE-311 (Missing Encryption of Sensitive Data)
Name:         Server Listens on Plain HTTP Without TLS
Severity:     Critical
File:         src/main.cpp, line 146 and 440
Description:  The server is created as httplib::Server (not httplib::SSLServer)
              and listens on plain HTTP. All traffic including JWT tokens,
              passwords, user credentials, and audio streams are transmitted in
              cleartext. The server binds to 0.0.0.0 (all interfaces) by default.
Exploitation: An attacker on the same network can perform passive eavesdropping
              or man-in-the-middle attacks to capture user passwords, JWT tokens,
              and audio content. All authentication is effectively bypassed by
              network sniffing.

VULNERABILITY #2
================================================================================
CWE ID:       CWE-798 (Use of Hard-Coded Credentials / Secrets in Config)
Name:         JWT Secret Loaded from Plaintext Config Without Validation
Severity:     High
File:         src/app_config.cpp, line 38
Description:  The JWT secret is read directly from a JSON configuration file with
              no validation of its strength or entropy. There is no minimum
              length requirement and no fallback to environment variables. If the
              config file ships with a weak or default secret, all JWT tokens
              become trivially forgeable.
Exploitation: An attacker who guesses or brute-forces a weak JWT secret can forge
              arbitrary JWT tokens, impersonate any user (including artists), and
              gain full access to all API endpoints.

VULNERABILITY #3
================================================================================
CWE ID:       CWE-522 (Insufficiently Protected Credentials)
Name:         JWT Token Accepted via URL Query Parameter
Severity:     High
File:         src/main.cpp, lines 97-99
Description:  The authenticate_request() function accepts JWT tokens from the
              "token" query parameter as a fallback. This is used on line 347
              where stream URLs embed the token in the query string:
              song_json["streamUrl"] = "/api/songs/" + song.id + "/stream?token=" + query_token;
Exploitation: Tokens in URLs are logged by web servers, stored in browser
              history, cached by proxies, and visible in Referer headers. An
              attacker who gains access to server logs, proxy logs, or browser
              history can steal valid JWT tokens and hijack user sessions.

VULNERABILITY #4
================================================================================
CWE ID:       CWE-74 (Injection in HTTP Header)
Name:         Unsanitized Filename in Content-Disposition Header
Severity:     High
File:         src/main.cpp, line 394
Description:  The song file_name is inserted directly into the Content-Disposition
              header: res.set_header("Content-Disposition", "inline; filename=\"" + song->file_name + "\"");
              The file_name field is loaded from the JSON catalog file. If the
              catalog is tampered with or a future change allows user-controlled
              filenames, characters like quotes or newlines could inject
              additional HTTP headers (HTTP Response Splitting).
Exploitation: If a filename contains \r\n characters, an attacker could inject
              arbitrary HTTP response headers, potentially enabling cache
              poisoning, session fixation, or XSS via injected headers.

VULNERABILITY #5
================================================================================
CWE ID:       CWE-287 (Improper Authentication - TOCTOU Race Condition)
Name:         Race Condition in User Registration (TOCTOU)
Severity:     Medium
File:         src/main.cpp, lines 189-206
Description:  The registration endpoint first checks if a user exists via
              user_store.find_by_email(email) (line 189) and then creates the
              user via user_store.create_user() (line 203). Between these two
              operations, another concurrent request could register the same
              email, creating a Time-of-Check-Time-of-Use race condition.
Exploitation: Concurrent registration requests with the same email could cause
              inconsistent behavior or duplicate accounts, depending on timing
              and the secondary Firestore-level checks.
