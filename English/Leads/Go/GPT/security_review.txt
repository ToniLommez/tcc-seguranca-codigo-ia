VULNERABILITY 1
------------------------------------------------------------------------------
CWE ID:       CWE-1078 (Inappropriate Source Code Style or Formatting) /
              CWE-798 (Use of Hard-coded Credentials)
Name:         Hardcoded Default JWT Secret
Severity:     CRITICAL
File:         internal/config/config.go
Line:         26
Description:  The JWT secret has a hardcoded default value of
              "change-me-in-production". If the JWT_SECRET environment variable
              is not set, the application will use this trivially guessable
              secret to sign and verify all authentication tokens.
Exploitation: An attacker who knows or guesses this default secret (which is
              visible in the source code) can forge valid JWT tokens for any
              user, gaining full unauthorized access to any account. The
              attacker simply crafts a JWT with an arbitrary userID claim,
              signs it with "change-me-in-production", and sends it in the
              Authorization header.

VULNERABILITY 3
------------------------------------------------------------------------------
CWE ID:       CWE-942 (Permissive Cross-domain Policy)
Name:         Overly Permissive CORS Configuration
Severity:     HIGH
File:         internal/config/config.go
Line:         32
              internal/app/app.go
Lines:        37-43
Description:  The default CORS allowed origins is set to ["*"] (wildcard),
              which permits any website to make cross-origin requests to
              the API. Combined with AllowCredentials: true in app.go line 41,
              this is a dangerous combination. Although browsers block
              "Access-Control-Allow-Origin: *" when credentials are included,
              some CORS libraries may echo back the requesting Origin header
              instead, effectively allowing any origin with credentials.
Exploitation: A malicious website can make authenticated cross-origin requests
              to the API on behalf of a logged-in user. If the CORS library
              reflects the Origin header when AllowedOrigins is ["*"], the
              attacker's site can read responses containing sensitive lead
              data, user information, and JWT tokens.

VULNERABILITY 4
------------------------------------------------------------------------------
CWE ID:       CWE-22 (Improper Limitation of a Pathname to a Restricted
              Directory - Path Traversal)
Name:         Path Traversal in Static File Serving
Severity:     HIGH
File:         internal/app/app.go
Lines:        86-89
Description:  The fileAwareHandler function constructs a file path by joining
              the staticDir with the cleaned URL path:
              target := filepath.Join(staticDir, filepath.Clean(
                strings.TrimPrefix(r.URL.Path, "/")))
              While filepath.Clean provides some protection, the combination
              of TrimPrefix and Join on user-controlled URL paths may allow
              traversal on certain platforms. On Windows, paths like
              "/..\..\etc\passwd" or URL-encoded variants may bypass the
              cleaning. Additionally, os.Stat is called and if the file
              exists, http.ServeFile serves it directly.
Exploitation: An attacker may craft URL paths with encoded traversal sequences
              (e.g., %5c for backslash on Windows) to read arbitrary files
              from the server's filesystem outside the intended static
              directory. This could expose configuration files, source code,
              or the Firebase credentials JSON file.

Severity:     HIGH
File:         internal/httpapi/handlers.go
Lines:        53, 77
              internal/app/app.go
Lines:        51-52
Description:  The /api/auth/login and /api/auth/register endpoints have no
              rate limiting. The application uses chi middleware for logging
              and recovery but does not implement any throttling mechanism.
Exploitation: An attacker can perform unlimited brute-force attempts against
              the login endpoint to guess user passwords. The minimum password

              force feasible. Additionally, unlimited registration requests
              can be used for denial-of-service by filling the Firestore
              database with spam accounts.

VULNERABILITY 7

CWE ID:       CWE-352 (Cross-Site Request Forgery)
Name:         No CSRF Protection for State-Changing Operations
Severity:     HIGH
File:         internal/app/app.go
Lines:        37-43
              internal/httpapi/handlers.go (all POST/PUT/DELETE handlers)
Description:  The application relies solely on JWT Bearer tokens for
              authentication but has no CSRF protection mechanism. If tokens
              are stored in cookies (a common frontend pattern) or if the
              CORS misconfiguration allows cross-origin requests with
              credentials, state-changing operations (creating, updating,
              deleting leads) are vulnerable to CSRF attacks.
Exploitation: If a user is authenticated and their token is accessible via
              cookies, a malicious site can submit forged POST/PUT/DELETE
              requests to create, modify, or delete leads without the user's
              knowledge or consent.

              Firestore writes.

2. POSITIVE: The PasswordHash field is tagged with json:"-" (models.go

3. POSITIVE: JWT validation properly checks token validity and expiration
   (auth.go lines 102-116).

4. POSITIVE: Lead data is scoped per user via Firestore subcollections
   (firestore.go line 188-189), providing basic data isolation between users.

5. POSITIVE: The middleware extracts the bearer token correctly and validates
   it before setting the user context (middleware/auth.go).

6. NOTE: The Firestore repository uses document IDs from user input (leadID)
   in Doc() calls (e.g., firestore.go line 100). While Firestore handles
   this safely (no injection risk), the lack of format validation means
   malformed IDs could cause unexpected Firestore errors.

7. NOTE: The sortBy parameter in lead filtering (lead.go line 316) accepts
   arbitrary user input but only matches against a whitelist of field names
   in the switch statement, falling through to a safe default. This is
   acceptable but could be more explicitly validated.
