VULNERABILITY #1
==============================================================================
CWE ID: CWE-798
Name: Hardcoded Cryptographic Secret
Severity: Critical
File: backend/config.h
Line: 8
Description: The JWT signing secret is hardcoded directly in the source code
as "music-stream-ultra-secret-key-change-in-prod-2024". This secret is
compiled into the binary and visible to anyone with access to the source code
or who can extract strings from the compiled binary.
Exploitation: An attacker who obtains this secret (via source code leak,
binary reverse engineering, or repository access) can forge arbitrary JWT
tokens, impersonate any user or artist, and gain full unauthorized access to
all API endpoints.

VULNERABILITY #2
==============================================================================
CWE ID: CWE-22
Name: Path Traversal in Static File Serving
Severity: Critical
File: backend/main.cpp
Lines: 89-106
Description: The static file serving routes construct file paths by directly
concatenating user-supplied URL path segments with the FRONTEND_DIR base
directory. The regex patterns do not filter out directory traversal sequences
such as "..". An attacker can craft a URL like "/../../../etc/passwd.html" (if
the server decodes the URL before matching) to read arbitrary files from the
server's filesystem, as long as the filename ends with one of the allowed
extensions.
Exploitation: An attacker sends a request such as
GET /../../backend/config.h.html to read sensitive source files, configuration,
or the Firebase credentials JSON file from the server filesystem.

VULNERABILITY #3
==============================================================================
CWE ID: CWE-942
Name: Overly Permissive CORS Policy
Severity: High
File: backend/main.cpp
Lines: 24-28
Description: The CORS policy is set to "Access-Control-Allow-Origin: *",
which allows any website in the world to make cross-origin requests to this
API. This is applied to ALL responses, including authenticated API endpoints.
Exploitation: A malicious website can make authenticated API requests on
behalf of a logged-in user if their browser has stored credentials or tokens.
This enables cross-site attacks where an attacker's page makes requests to
the music streaming API using the victim's session.

VULNERABILITY #4
==============================================================================
CWE ID: CWE-208
Name: Timing-Vulnerable JWT Signature Comparison
Severity: High
File: backend/auth/jwt_helper.cpp
Line: 132
Description: The JWT signature verification uses a simple string comparison
operator (!=) to compare the expected signature with the provided signature:
"if (expected_sig_enc != sig_enc) return false;". Standard string comparison
is not constant-time and can leak information about how many characters of the
signature matched, enabling a timing side-channel attack.
Exploitation: An attacker can iteratively guess the correct HMAC signature
byte-by-byte by measuring response time differences. Each correct byte will
cause the comparison to take marginally longer before failing, allowing the
attacker to forge valid JWT tokens without knowing the secret key.

VULNERABILITY #5
==============================================================================
CWE ID: CWE-319
Name: No HTTPS/TLS Enforcement
Severity: Medium
File: backend/main.cpp
Lines: 59, 112
Description: The server uses httplib::Server (plain HTTP) rather than
httplib::SSLServer (HTTPS). All communication including JWT tokens,
passwords, and user credentials are transmitted in cleartext over the
network. The code defines CPPHTTPLIB_OPENSSL_SUPPORT at line 1 but never
uses it for the server.
Exploitation: An attacker on the same network (e.g., public Wi-Fi) can
intercept all traffic using passive sniffing, capturing JWT tokens,
plaintext passwords during login, and user data. This enables full account
takeover via token theft.
