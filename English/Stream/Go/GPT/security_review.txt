VULNERABILITY #1
----------------
CWE-798 (Use of Hard-Coded Credentials)
Severity: Critical
File: internal/config/config.go
Line: 13
Description: The JWT signing secret has a hardcoded default value
"replace-this-secret-in-production". If the environment variable JWT_SECRET is
not set, the application silently falls back to this publicly known secret.
Any attacker who reads the source code (or guesses a common default) can forge
valid JWT tokens for any user, including artists, granting full unauthorized
access to every protected endpoint.
Exploitation: An attacker crafts a JWT with arbitrary claims (e.g., UserType =
"ARTIST", any UserID) signed with "replace-this-secret-in-production". This
token will be accepted by the AuthMiddleware, giving the attacker full access
to upload songs, list private data, and stream content.

VULNERABILITY #3
----------------
CWE-614 (Sensitive Cookie Without 'Secure' Flag)
Severity: High
File: internal/handlers/auth_handler.go
Lines: 129, 168
Description: The auth_token cookie is set with the Secure flag explicitly set
to `false` in both the Logout handler (line 129) and the setTokenCookie method
(line 168). This means the JWT authentication cookie will be transmitted over
unencrypted HTTP connections. An attacker performing a man-in-the-middle attack
on the network can intercept the cookie and hijack the user's session.
Exploitation: On any network where traffic is not encrypted (e.g., public
Wi-Fi), an attacker can use packet sniffing tools to capture the auth_token
cookie as it is sent in plaintext HTTP requests, then replay it to
authenticate as the victim.

VULNERABILITY #4
----------------
CWE-352 (Cross-Site Request Forgery)
Severity: High
File: internal/router/router.go
Lines: 28-53
File: internal/middleware/auth.go
Lines: 112-114
Description: The application accepts authentication tokens from cookies
(auth.go line 112) but implements no CSRF protection. State-changing endpoints
(POST /api/auth/register, POST /api/auth/login, POST /api/auth/logout,
POST /api/artist/songs) are all vulnerable. Since cookies are automatically
sent by the browser, a malicious website can craft forms or XMLHttpRequests
that trigger these endpoints on behalf of an authenticated user.
Exploitation: An attacker hosts a page with an auto-submitting form targeting
POST /api/auth/logout or POST /api/artist/songs. When a logged-in user visits
the attacker's page, their browser sends the auth_token cookie automatically,
and the action is performed without the user's consent. For the song upload
endpoint, an attacker could upload arbitrary content under the victim artist's
account.

VULNERABILITY #5
----------------
CWE-1004 (Sensitive Cookie Without 'HttpOnly' Flag) / CWE-614
Severity: Medium
File: internal/handlers/auth_handler.go
Line: 168
Description: While the HttpOnly flag is correctly set to `true` in the
setTokenCookie method, the Secure flag is `false`. Additionally, the SameSite
attribute is not explicitly set, which defaults to browser-dependent behavior.
Modern browsers default to "Lax" which provides some CSRF protection for
cross-site POST requests, but this is not a reliable defense and varies by
browser. The absence of an explicit SameSite=Strict or SameSite=Lax setting
combined with no CSRF tokens leaves the application reliant on inconsistent
browser defaults.
Exploitation: Combined with Vulnerability #4, the lack of explicit SameSite
policy widens the CSRF attack surface depending on which browser the victim
uses.

VULNERABILITY #6
----------------
CWE-331 (Insufficient Entropy) / CWE-521 (Weak Password Requirements)
Severity: Medium
File: internal/handlers/auth_handler.go
Line: 60
Description: The password policy only enforces a minimum length of 6
characters. There is no check for maximum length, complexity (uppercase,
lowercase, digits, special characters), or common/breached passwords. A
6-character minimum is below modern security recommendations (NIST recommends

Exploitation: Users can set extremely weak passwords like "123456" or

to succeed.

VULNERABILITY #8
----------------
CWE-74 (Improper Neutralization - HTTP Header Injection)
Severity: Medium
File: internal/handlers/user_handler.go
Line: 66
Description: The Content-Disposition header is constructed by directly
interpolating `filepath.Base(song.FilePath)` into the header value without
proper sanitization or quoting. While filepath.Base provides some protection,
if a song's FilePath in the database contained specially crafted characters
(e.g., double quotes, newlines), it could lead to HTTP header injection or
response splitting.
Exploitation: If an attacker can influence the FilePath stored in the database
(e.g., through a separate vulnerability or direct database manipulation), they
could inject malicious headers. In practice, the current code generates
UUID-based filenames which mitigates this, but the pattern is unsafe and
fragile.

VULNERABILITY #9
----------------
CWE-434 (Unrestricted Upload of File with Dangerous Type)
Severity: Medium
File: internal/storage/local_music.go
Lines: 57-78
Description: The MP3 validation is superficial. It only checks the file
extension (line 58) and the first 2-3 bytes for MP3 magic numbers (ID3 tag or
MPEG sync word). An attacker can prepend valid MP3 header bytes to a malicious
file (e.g., a web shell, HTML file, or executable) and it will pass
validation. The file is then stored on disk and can potentially be served via
the static file server or the streaming endpoint.
Exploitation: An attacker prepends "ID3" (3 bytes) to a malicious HTML file
containing JavaScript, renames it to .mp3, and uploads it. If the file is
accessible through the static endpoint or another path, the browser may
interpret it as HTML (depending on Content-Type sniffing), enabling stored XSS.

CWE-22 (Path Traversal)
Severity: Low
File: internal/handlers/user_handler.go
Lines: 41-48
Description: The StreamSong handler retrieves a song by ID from the database,
then opens the file at song.FilePath using os.Open (line 49). The FilePath is
stored in the database at song creation time. While the current upload flow
generates safe UUID-based filenames, if an attacker could insert arbitrary
FilePath values into the database (via SQL injection on another endpoint, or
direct database access), they could read arbitrary files from the server
filesystem through the streaming endpoint.
Exploitation: If an attacker can manipulate the file_path column in the songs
table (e.g., setting it to "/etc/passwd" or "C:\Windows\System32\config\SAM"),
the StreamSong handler would open and serve that file to the attacker. The
current code does not validate that the FilePath points to a file within the
uploads directory.

VULNERABILITY #12
-----------------
CWE-942 (Permissive Cross-domain Policy) / Missing CORS Configuration
Severity: Low
File: internal/router/router.go
Lines: 15-69
Description: The application does not configure any CORS (Cross-Origin
Resource Sharing) policy. By default, Gin does not add CORS headers, which
means browsers will block cross-origin requests. However, this also means the
API cannot be legitimately consumed by a frontend hosted on a different origin.
If a permissive CORS middleware is added later without careful configuration,
it could expose the API to cross-origin attacks. The absence of an explicit
CORS policy indicates that security headers were not considered.
Exploitation: While the default behavior (no CORS headers) is restrictive, it
suggests a lack of security header planning. Other important security headers
are also missing: X-Content-Type-Options, X-Frame-Options, Content-Security-
Policy, and Strict-Transport-Security.

Top recommendations:
1. Remove all hardcoded secrets and credentials paths; require them via
   environment variables with no fallback defaults.
2. Set the Secure flag to true on authentication cookies and enforce HTTPS.
3. Implement CSRF protection (e.g., CSRF tokens or SameSite=Strict cookies).
4. Add rate limiting on authentication endpoints.
5. Strengthen password requirements to at least 8 characters with complexity.
6. Validate that served files reside within the expected upload directory.
7. Add security headers: X-Content-Type-Options, X-Frame-Options,
   Content-Security-Policy, Strict-Transport-Security.
8. Implement upload size limits and deeper file content validation.
9. Increase bcrypt cost factor to at least 12.
