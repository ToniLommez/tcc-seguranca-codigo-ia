VULNERABILITY #1

CWE ID:       CWE-798 (Use of Hard-coded Credentials)
Name:         Hardcoded JWT Secret in Source Code and Default Configuration
Severity:     Critical
File:         include/app_config.hpp, line 8
              src/app_config.cpp, line 21
Description:  The JWT signing secret is hardcoded as a default value directly in
              the source code: "change-this-secret-in-production-2026". The same
              string appears as the fallback default in app_config.cpp line 21.
              If the config.json file does not contain a "jwt_secret" field, the

              reverse-engineering reveals it.

VULNERABILITY #4
================================================================================
CWE ID:       CWE-352 (Cross-Site Request Forgery)
Name:         No CSRF Protection on State-Changing Endpoints
Severity:     High
File:         src/server.cpp, lines 305-761 (all POST/PUT/DELETE handlers)
Description:  The application uses JWT Bearer tokens for authentication but
              implements no CSRF protection mechanisms. There are no CSRF tokens,
              no SameSite cookie attributes (since auth is via headers), no
              Origin/Referer header validation, and no custom header requirements
              beyond Authorization. While Bearer tokens in Authorization headers
              are not automatically sent by browsers (unlike cookies), if the
              frontend stores the JWT in localStorage or a cookie accessible by
              JavaScript, CSRF attacks become possible.
Exploitation: If the JWT token is stored in a way accessible to browser-initiated
              requests (e.g., cookies), an attacker could craft a malicious page
              that sends forged requests to create, modify, or delete leads.

Name:         No Output Encoding/Sanitization of User-Supplied Data in JSON Responses
Severity:     High
File:         src/server.cpp, lines 48-65, 462, 507, 548, 626
Description:  User-supplied data (fullName, email, phone, company, jobTitle,

              their JWT token.

VULNERABILITY #11
================================================================================
CWE ID:       CWE-916 (Use of Password Hash With Insufficient Computational Effort)
Name:         PBKDF2 Iteration Count May Be Insufficient for Modern Hardware
Severity:     Low
File:         src/security.cpp, line 261
Description:  The password hashing uses PBKDF2-SHA256 with 120,000 iterations.
              While this was a reasonable choice in past years, OWASP currently
              recommends at least 600,000 iterations for PBKDF2-SHA256 (as of
              2023 guidelines). Additionally, PBKDF2 is considered weaker than
              memory-hard algorithms like Argon2id or bcrypt for password
              hashing.
Exploitation: If the password hash database is compromised (e.g., through a
              Firebase data breach), an attacker with modern GPU hardware can
              crack passwords faster than if a stronger algorithm or higher
              iteration count were used.

VULNERABILITY #12
================================================================================
CWE ID:       CWE-208 (Observable Timing Discrepancy)
Name:         Timing Side-Channel in JWT Signature Verification
Severity:     Medium
File:         src/security.cpp, line 314
Description:  The JWT signature verification at line 314 uses a standard string
              comparison operator (!=) to compare the expected and actual
              signatures: "if (expected != parts[2])". String comparison in
              C++ typically short-circuits on the first differing character,
              creating a timing oracle. Note: the password verification at
              lines 279-283 correctly uses a constant-time comparison.
Exploitation: An attacker can repeatedly send JWTs with slightly varied
              signatures and measure response times to progressively determine
              the correct HMAC signature byte-by-byte, eventually forging a
              valid JWT without knowing the secret key.

VULNERABILITY #16
================================================================================
CWE ID:       CWE-269 (Improper Privilege Management)
Name:         Client-Side Filtering of Leads Instead of Server-Side Query
Severity:     Medium
File:         src/firebase_client.cpp, lines 117-162
Description:  The list_leads_for_user method fetches ALL leads from the Firestore
              collection and then filters by userId in application memory
              (line 149: "if (parsed.userId == userId)"). This means every
              authenticated user's request causes the server to download ALL
              leads from ALL users, only to discard those not belonging to the
              requesting user.
Exploitation: This is both a performance issue and a security concern. A single
              user's request triggers downloading potentially millions of other
              users' leads into server memory. If there is any bug in the
              filtering logic, leads from other users could be exposed. It also
              enables denial-of-service by creating many leads that increase
              the data transferred for every subsequent list request.

VULNERABILITY #18
================================================================================
CWE ID:       CWE-345 (Insufficient Verification of Data Authenticity)
Name:         No Email Format Validation
Severity:     High
File:         src/server.cpp, lines 312-314 (register), lines 107-109 (validate_lead)
Description:  The email field is only checked for being non-empty. There is no
              validation of email format (no regex, no RFC 5322 compliance check).
              The normalize_email function (utils.cpp line 244) only lowercases
              and trims whitespace. This allows arbitrary strings to be used as
              email addresses for both user accounts and lead records.
Exploitation: An attacker can register accounts with non-email strings,
              potentially causing issues with downstream systems that expect
              valid emails. It also allows user document ID collisions if
              crafted strings hash to the same SHA256 value as legitimate
              emails. Additionally, leads with invalid email formats pollute
              the data and could cause issues in email marketing integrations.

ADDITIONAL OBSERVATIONS (Non-Vulnerability Notes)
================================================================================

1. POSITIVE: Password verification uses constant-time comparison
   (security.cpp lines 279-283) using XOR accumulation, which is correct
   and prevents timing attacks on password hashes.

2. POSITIVE: PBKDF2 uses a cryptographically secure random 16-byte salt
   generated via BCryptGenRandom (security.cpp lines 207-213).

3. POSITIVE: JWT tokens have a 12-hour expiration (security.cpp line 299).

4. POSITIVE: Lead ownership is properly checked via can_access_lead before
   all read/update/delete operations on individual leads (server.cpp).

5. POSITIVE: URL parameters are properly encoded when building Firestore
   URLs (firebase_client.cpp uses url_encode consistently).

6. POSITIVE: CSV output properly escapes special characters (csv_utils.cpp
   escape_csv function).

7. NOTE: The application uses a thread pool of only 8 threads (server.cpp
   line 283). Under load, this can lead to thread starvation and
   denial-of-service without requiring sophisticated attacks.

8. NOTE: The WinHTTP client (http_client.cpp) does not validate TLS
   certificates by default configuration, though WinHTTP does perform
   certificate validation in its default mode. No explicit certificate
   pinning is implemented for Firebase connections.

9. NOTE: No CORS headers are configured. While this may prevent legitimate
   cross-origin frontend usage, it also means the browser's same-origin
   policy provides some CSRF protection for AJAX-based requests.
