EXECUTIVE SUMMARY

Four vulnerabilities were retained as the most relevant: a hardcoded JWT

DETAILED FINDINGS
================================================================================

------------------------------------------------------------------------------
VULNERABILITY #1
------------------------------------------------------------------------------
CWE ID:        CWE-798 (Use of Hard-coded Credentials)
Name:          Hardcoded Default JWT Secret
Severity:      CRITICAL
File:          utils/jwt.go
Line(s):       61-67
Description:
  The function getJWTSecret() falls back to a hardcoded default secret string
  "lead-manager-default-secret-change-in-production" when the JWT_SECRET
  environment variable is not set. This value is embedded directly in the
  source code and is publicly visible to anyone with access to the repository.
Exploitation:
  If the application is deployed without setting the JWT_SECRET environment
  variable, any attacker who knows or guesses this default string can forge
  valid JWT tokens with an arbitrary userID claim and gain full access to any
  user's data without authentication.

------------------------------------------------------------------------------
VULNERABILITY #2
------------------------------------------------------------------------------
CWE ID:        CWE-942 (Permissive Cross-domain Policy)
Name:          Overly Permissive CORS Configuration (AllowOrigins: *)
Severity:      HIGH
File:          main.go
Line(s):       36-42
Description:
  The CORS middleware is configured with AllowOrigins set to ["*"], allowing
  any origin to make cross-origin requests to the API. All HTTP methods are
  allowed from any origin, and the Authorization header is exposed cross-origin.
Exploitation:
  An attacker can host a malicious website that makes authenticated API calls
  to the Lead Manager backend using a victim's JWT token. Since the server
  accepts requests from any origin, the attacker's site can read response data,
  export all leads, modify data, or delete records.

------------------------------------------------------------------------------
VULNERABILITY #3
------------------------------------------------------------------------------
CWE ID:        CWE-352 (Cross-Site Request Forgery)
Name:          No CSRF Protection on State-Changing Endpoints
Severity:      HIGH
File:          main.go
Line(s):       52-84
Description:
  The application relies solely on JWT bearer tokens in the Authorization
  header for authentication. The wildcard CORS policy combined with the
  AllowHeaders including "Authorization" means that any malicious website can
  craft cross-origin requests with the token if it is stored in localStorage
  (common for SPAs). There is no CSRF token, no SameSite cookie policy, and no
  Origin/Referer validation.
Exploitation:
  If an attacker obtains access to the JWT token (e.g., via XSS on the
  frontend), they can make state-changing requests (create, update, delete
  leads) from any origin. The wildcard CORS makes this trivial.

------------------------------------------------------------------------------
VULNERABILITY #4
------------------------------------------------------------------------------
CWE ID:        CWE-311 (Missing Encryption of Sensitive Data)
Name:          No HTTPS/TLS Configuration
Severity:      MEDIUM
File:          main.go
Line(s):       97
Description:
  The server starts with r.Run() which listens on plain HTTP. There is no
  TLS configuration, no HSTS headers, and no redirect from HTTP to HTTPS.
  JWT tokens and user credentials (passwords) are transmitted in plaintext.
Exploitation:
  An attacker performing a man-in-the-middle attack (e.g., on a shared
  network) can intercept login credentials (email + password) and JWT tokens
  in transit, gaining full access to the victim's account.
