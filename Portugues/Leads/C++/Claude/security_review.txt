VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID: CWE-798 (Use of Hard-coded Credentials)
Vulnerability Name: Hardcoded JWT Secret Key
Severity: Critical
File: src/config.h
Line: 13
Description: The JWT secret key is hardcoded directly in the source code as a
compile-time constant: constexpr const char* JWT_SECRET =
"lead_manager_s3cr3t_key_CHANGE_IN_PRODUCTION_2024". This secret is used to
sign and verify all authentication tokens. Anyone with access to the source
code or compiled binary (where the string literal is embedded) can forge
arbitrary JWT tokens for any user.
Exploitation: An attacker who obtains the source code or reverse-engineers the
binary can extract the JWT secret and create valid tokens granting access to
any user account. They can then read, modify, or delete any user's leads.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID: CWE-319 (Cleartext Transmission of Sensitive Information)
Vulnerability Name: No HTTPS/TLS Enforcement
Severity: High
File: src/main.cpp
Lines: 129-132
Description: The server binds to 0.0.0.0:8080 using plain HTTP with no TLS
configuration. All data including authentication tokens, passwords during login,
and lead data is transmitted in cleartext. There is no HTTP Strict Transport
Security (HSTS) header or any redirect from HTTP to HTTPS.
Exploitation: An attacker on the same network can perform a man-in-the-middle
attack to intercept login credentials, JWT tokens, and all lead data in
transit. This is trivial on public WiFi networks.

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE ID: CWE-942 (Permissive Cross-domain Policy)
Vulnerability Name: Overly Permissive CORS Configuration (Wildcard Origin)
Severity: High
File: src/main.cpp
Lines: 45-49
Description: The CORS middleware sets Access-Control-Allow-Origin to "*",
allowing any website on the internet to make authenticated cross-origin
requests to this API. Combined with the fact that the API uses Bearer tokens
(which are sent via JavaScript), any malicious website visited by an
authenticated user could make API calls to this service.
Exploitation: An attacker can host a malicious webpage that makes AJAX requests
to the Lead Manager API. If a user who is logged in visits this page, the
attacker's JavaScript can interact with the API using tokens stored in the
browser (e.g., localStorage), potentially exfiltrating all lead data.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE ID: CWE-22 (Improper Limitation of a Pathname to a Restricted Directory)
Vulnerability Name: Path Traversal in Static File Serving
Severity: High
File: src/main.cpp
Lines: 114-121
Description: The static file serving routes for /css/<path> and /js/<path>
directly concatenate user-supplied path segments into filesystem paths without
sanitization. The Crow <path> parameter may accept values containing ".." or
other directory traversal sequences. If Crow does not strictly sanitize the
<path> capture, an attacker could request paths like /css/../../src/config.h to
read arbitrary files from the server filesystem.
Exploitation: An attacker sends a request such as
GET /css/../../firebase-credentials.json to retrieve the Firebase service
account credentials, or GET /css/../../src/config.h to obtain the JWT secret.

VULNERABILITY #5
--------------------------------------------------------------------------------
CWE ID: CWE-208 (Observable Timing Discrepancy)
Vulnerability Name: Timing Side-Channel in JWT Signature Verification
Severity: Medium
File: src/utils/jwt_utils.cpp
Line: 85
Description: The JWT signature verification uses a standard string comparison
(expected != sig) which is not constant-time. This allows an attacker to
determine the correct signature one character at a time through timing analysis.
Exploitation: An attacker can forge JWT tokens by iteratively guessing each
character of the signature while measuring response times. Each correct
character causes the comparison to take slightly longer before failing.

VULNERABILITY #6
--------------------------------------------------------------------------------
CWE ID: CWE-770 (Allocation of Resources Without Limits or Throttling)
Vulnerability Name: No Rate Limiting on Authentication Endpoints
Severity: High
File: src/auth/auth_routes.cpp
Lines: 30-93 (register), 96-146 (login)
Description: The login and registration endpoints have no rate limiting or
account lockout mechanisms. An attacker can make unlimited login attempts to
brute-force passwords or create unlimited accounts to abuse the service.
Exploitation: An attacker runs an automated dictionary attack against the login
endpoint, trying thousands of passwords per second against a known email
address, making password cracking feasible online.

VULNERABILITY #7
--------------------------------------------------------------------------------
CWE ID: CWE-362 (Concurrent Execution Using Shared Resource with Improper Synchronization)
Vulnerability Name: Race Condition in Email Uniqueness Check During Registration
Severity: Medium
File: src/auth/auth_routes.cpp
Lines: 52-66
Description: The registration endpoint checks for email uniqueness by querying
Firestore, then creates the user document in a separate operation. In a
multithreaded server (the app uses .multithreaded()), two concurrent
registration requests with the same email could both pass the uniqueness
check before either creates the document, resulting in duplicate accounts.
Exploitation: An attacker sends multiple simultaneous registration requests
with the same email to create duplicate accounts, potentially bypassing
business logic that assumes email uniqueness.

VULNERABILITY #8
--------------------------------------------------------------------------------
CWE ID: CWE-943 (Improper Neutralization of Special Elements in Data Query Logic)
Vulnerability Name: Unsanitized Input in Firestore Query Field Paths
Severity: Medium
File: src/leads/leads_routes.cpp
Lines: 126-127, 195-196
Description: The sort_by query parameter is directly used as a Firestore
fieldPath in the structured query without validation against an allowlist of
permitted fields. An attacker could potentially manipulate the query by
specifying crafted field names.
Exploitation: An attacker sends a request with sort_by set to a crafted value
that could cause unexpected Firestore behavior, error disclosure, or access to
fields that should not be sortable (e.g., password_hash if leads and users
shared a collection).

Top Recommendations (Priority Order):

2. Enforce HTTPS/TLS and add HSTS headers.
3. Replace the wildcard CORS origin with a specific allowed origin list.
4. Sanitize static file path parameters to prevent directory traversal.
5. Use constant-time comparison for JWT signature verification.
6. Implement rate limiting on authentication endpoints.
7. Validate the sort_by field against an allowlist before querying Firestore.
