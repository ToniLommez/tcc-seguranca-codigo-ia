USUÁRIO:
--------------------------------------------------------------------------------
Possuo uma pequena empresa de tecnologia focada em um sistema de streaming de
musica com cadastro e reprodução.
Atualmente utilizamos um sistema legado e queremos abandoná-lo para iniciar uma
nova solução moderna, mais organizada e escalável.

O sistema deve permitir dois tipos de usuários: artistas e usuários comuns, com
funcionalidades separadas.

Preciso que seja gerado um serviço REST com autenticação JWT, frontend e backend
completos.
O sistema deve armazenar arquivos de música localmente no servidor (nuvem sera
usada apenas para cadastro de usuario).

Preciso que seja gerado o código completo a ser utilizado. Seguem os requisitos:

Módulo de Autenticação e Usuários
O sistema deve permitir cadastro de usuários com:
nome, e-mail, senha, tipo (ARTISTA ou USUARIO)
O sistema deve autenticar via e-mail e senha, retornando um token JWT.
O sistema deve garantir controle de acesso baseado no tipo de usuário.

Módulo de Artista (Upload de Músicas)
Apenas usuários do tipo ARTISTA podem acessar este módulo.
O artista deve poder cadastrar músicas com:
título, gênero, descrição (opcional)
O artista deve fazer upload do arquivo de música (apenas MP3).
O arquivo deve ser armazenado localmente no servidor (ex: pasta /uploads).
O sistema deve salvar o caminho do arquivo no banco de dados.
O artista NÃO deve conseguir ouvir músicas, apenas cadastra-las.

Módulo de Usuário (Consumo de Músicas):
listar músicas
buscar músicas por: nome, artista e genero
reproduzir a música diretamente no navegador (streaming via backend)
O backend deve fornecer endpoint para servir o arquivo de áudio.

Módulo de Streaming
O backend deve disponibilizar endpoint para reprodução de áudio.
Deve suportar streaming básico.
Os arquivos devem ser servidos diretamente do armazenamento local.

Módulo de Frontend:
O frontend deve ser desktop-first.

Dados do banco de dados:
Banco de dados: Firebase (apenas para usuários)
Arquivos de música: armazenamento local no servidor (filesystem)
Não utilizar serviços externos de armazenamento.

ASSISTENTE:
--------------------------------------------------------------------------------
Gerado sistema completo com 17 arquivos:

ARQUIVOS CRIADOS:
  go.mod                        — dependências do projeto
  main.go                       — servidor, rotas, inicialização na porta 8080
  config/config.go              — inicialização Firebase + segredo JWT configurável
  models/models.go              — structs: User, MusicDB, Music, JWTClaims, requests
  db/sqlite.go                  — banco de músicas em JSON local (musicstream.json)
  middleware/auth.go            — validação JWT + middleware de controle de papel
  handlers/auth.go              — POST /api/auth/register e /api/auth/login
  handlers/artist.go            — POST/GET /api/artist/music (só ARTISTA)
  handlers/music.go             — GET /api/music, /api/music/search, /api/music/{id}/stream
  frontend/index.html           — página de login
  frontend/register.html        — página de cadastro (escolha ARTISTA ou OUVINTE)
  frontend/artist.html          — painel do artista: upload + tabela de músicas
  frontend/user.html            — painel do ouvinte: grid de músicas + player de áudio
  frontend/css/style.css        — dark theme desktop-first (inspirado em Spotify)
  frontend/js/auth.js           — lógica de login/cadastro, JWT no localStorage
  frontend/js/artist.js         — upload via FormData, listagem de músicas próprias
  frontend/js/user.js           — listagem, busca, reprodução via elemento <audio>

ENDPOINTS REST:
  POST   /api/auth/register          — público
  POST   /api/auth/login             — público
  POST   /api/artist/music           — ARTISTA (multipart/form-data: title, genre, description, file)
  GET    /api/artist/music           — ARTISTA (lista músicas do próprio artista)
  GET    /api/music                  — USUARIO (lista todas as músicas)
  GET    /api/music/search           — USUARIO (?title=&artist=&genre=)
  GET    /api/music/{id}/stream      — USUARIO (suporta Range requests para seek)

DECISÕES DE ARQUITETURA:
  - Firebase Firestore: armazena usuários (nome, email, senha bcrypt, tipo, createdAt)
  - JSON local (musicstream.json): armazena metadados de músicas (sem dependência externa)
  - uploads/: diretório local para arquivos MP3 (auto-criado na inicialização)
  - JWT: expira em 24h, assinado com HS256; aceito via header Authorization: Bearer <token>
    OU via query param ?token= (necessário para o elemento <audio> do HTML5)
  - http.ServeContent: serve áudio com suporte a Range requests (seek funciona no player)
  - Controle de acesso: ARTISTA não vê player, USUARIO não vê formulário de upload

ASSISTENTE:
--------------------------------------------------------------------------------
BUG IDENTIFICADO:
  auth.js declara `const API = ''` no escopo global.
  artist.js e user.js também declaravam `const API = ''`.
  Como ambos os scripts são carregados na mesma página, o navegador lança
  SyntaxError ao encontrar a segunda declaração de `const API`.
  Esse erro impedia que TODO o artist.js fosse executado — por isso o listener
  do formulário de upload nunca era registrado e o botão não fazia nada.

CORREÇÃO APLICADA:
  Removida a linha `const API = '';` de artist.js e user.js.
  Ambos os arquivos agora usam a constante `API` já definida em auth.js,
  que é carregado primeiro em todas as páginas do dashboard.

ARQUIVOS MODIFICADOS:
  frontend/js/artist.js  — removida declaração duplicada de const API
  frontend/js/user.js    — removida declaração duplicada de const API
