VULNERABILITY #1
================================================================================
CWE ID:       CWE-798 (Use of Hard-coded Credentials)
Name:         Hardcoded JWT Secret Key
Severity:     Critical
File:         internal/config/config.go
Line:         33
Description:  The JWT signing secret has a hardcoded default value
              "troque-esta-chave-em-producao" (Portuguese for "change this key
              in production"). If the JWT_SECRET environment variable is not
              set, the application uses this weak, publicly visible secret to
              sign and verify all authentication tokens.
Exploitation: An attacker who knows this default secret (which is visible in
              the source code) can forge valid JWT tokens for any user,
              including fabricating arbitrary user IDs and emails. This grants
              full unauthenticated access to the entire application and all
              user data by crafting a token with jwt.SigningMethodHS256 and
              the known secret.

VULNERABILITY #3
================================================================================
CWE ID:       CWE-352 (Cross-Site Request Forgery)
Name:         Missing CSRF Protection on State-Changing API Endpoints
Severity:     High
File:         internal/http/server.go
Lines:        107-127
Description:  The application exposes POST, PUT, and DELETE API endpoints for
              creating, updating, and deleting leads and interactions. The
              authentication relies solely on cookies (with SameSite=Lax) and
              Bearer tokens. There is no CSRF token validation on any
              state-changing endpoint. While SameSite=Lax provides partial
              protection, it does not protect against top-level POST
              navigations from external sites.
Exploitation: An attacker can craft a malicious page that submits a form via
              POST to endpoints like /api/leads or /api/auth/logout. Since
              SameSite=Lax allows cookies on top-level navigations, the
              browser will include the authentication cookie. The attacker
              can create or delete leads, add interactions, or log out the
              user without their consent.

CWE ID:       CWE-614 (Sensitive Cookie Without 'Secure' Flag)
Name:         Authentication Cookie Missing Secure Flag
Severity:     High
File:         internal/http/server.go
Lines:        531-539
Description:  The setAuthCookie function creates an authentication cookie
              with HttpOnly and SameSite=Lax, but does NOT set the Secure
              flag. This means the cookie (containing the JWT token) will be
              transmitted over unencrypted HTTP connections.
Exploitation: An attacker performing a man-in-the-middle attack on any

              authentication cookie and hijack the user's session, gaining
              full access to their account and all lead data.

VULNERABILITY #5

CWE ID:       CWE-200 (Exposure of Sensitive Information)
Name:         JWT Token Returned in API Response Body
Severity:     High
File:         internal/http/server.go
Lines:        294-297, 317-320
Description:  Both handleRegisterAPI and handleLoginAPI return the raw JWT
              token in the JSON response body alongside the user object. The
              token is already set as an HttpOnly cookie, making the body
              inclusion unnecessary for browser-based clients. Exposing the
              token in the response body makes it accessible to JavaScript
              code, defeating the purpose of HttpOnly cookies.
Exploitation: If any XSS vulnerability exists (in this app or any app on the
              same origin), the attacker can read the token from the response
              body or from JavaScript variables that store it. The token can
              then be exfiltrated and used for session hijacking from a
              different machine.

Name:         Missing Rate Limiting on Authentication Endpoints
Severity:     Medium
File:         internal/http/server.go
Lines:        108-109
Description:  The /api/auth/login and /api/auth/register endpoints have no
              rate limiting. The application uses bcrypt for password hashing,
              which provides some protection against rapid brute-force, but
              there is no mechanism to limit the number of login attempts per
              IP address or per account.
Exploitation: An attacker can perform brute-force attacks against user
              accounts by sending unlimited login requests. Credential
              stuffing attacks using leaked password databases are also
              feasible. Additionally, repeated bcrypt computations from
              rapid login attempts could be used as a CPU-based denial of
              service attack.

VULNERABILITY #10
================================================================================
CWE ID:       CWE-942 (Permissive Cross-domain Policy)
Name:         Missing CORS Configuration
Severity:     Medium
File:         internal/http/server.go
Lines:        93-139
Description:  The router setup does not configure any CORS (Cross-Origin
              Resource Sharing) policy. Without an explicit CORS configuration,
              the browser's default same-origin policy applies, but the
              absence of explicit CORS headers means there is no defense-in-
              depth against misconfiguration. If a reverse proxy or CDN adds
              permissive CORS headers, the API would be exposed.
Exploitation: If the application is deployed behind infrastructure that adds
              overly permissive CORS headers (e.g., Access-Control-Allow-
              Origin: *), any malicious website could make authenticated API
              requests using the user's cookies and read the responses,
              including sensitive lead data and JWT tokens.
