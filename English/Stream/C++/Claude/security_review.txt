VULNERABILITY #1
--------------------------------------------------------------------------------
CWE ID: CWE-798 (Use of Hard-Coded Credentials)
Vulnerability Name: Hardcoded JWT Secret Key
Severity: Critical
File: src/main.cpp
Line: 19
Description: The JWT signing secret is hardcoded directly in the source code as a string literal: "soundstream-secret-key-change-in-production-2024". This secret is used to sign and verify all authentication tokens. Anyone with access to the source code (or the compiled binary via string extraction) can forge valid JWT tokens for any user, including impersonating artists or administrators.
Exploitation: An attacker who obtains the binary or source code can extract the secret string and craft arbitrary JWT tokens with any user_id, email, type, and name claims, granting them full access to the application as any user or role.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE ID: CWE-942 (Permissive Cross-domain Policy with Untrusted Domains)
Vulnerability Name: Overly Permissive CORS Configuration (Wildcard Origin)
Severity: High
File: src/main.cpp
Lines: 71-75
Description: The setup_cors function sets Access-Control-Allow-Origin to "*", allowing any website on the internet to make authenticated requests to the API. Combined with allowing the Authorization header, this enables any malicious website to interact with the API on behalf of authenticated users whose browsers visit the malicious site.
Exploitation: An attacker can create a malicious website that makes AJAX requests to the SoundStream API. If a logged-in user visits this malicious site, the attacker's JavaScript can perform actions on the API using the victim's credentials, such as uploading songs or accessing song lists.

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE ID: CWE-319 (Cleartext Transmission of Sensitive Information)
Vulnerability Name: Server Listens on HTTP Without TLS/SSL
Severity: High
File: src/main.cpp
Lines: 539-548
Description: The server listens on plain HTTP (0.0.0.0:8080) without any TLS/SSL encryption. All data transmitted between the client and server, including JWT tokens, user credentials, and audio streams, is sent in cleartext.
Exploitation: An attacker on the same network (e.g., public Wi-Fi, corporate LAN) can perform a man-in-the-middle attack to intercept login credentials, steal JWT tokens, or eavesdrop on audio streams. Captured JWT tokens can be reused to impersonate users.

VULNERABILITY #4
--------------------------------------------------------------------------------
CWE ID: CWE-200 (Exposure of Sensitive Information to an Unauthorized Actor)
Vulnerability Name: Token Accepted via Query Parameter
Severity: High
File: src/main.cpp
Lines: 57-59
Description: The authenticate function accepts JWT tokens via the "token" query parameter as a fallback when no Authorization header is present. Query parameters are logged in web server access logs, proxy logs, browser history, and HTTP Referer headers, exposing the authentication token in multiple locations.
Exploitation: An attacker with access to server logs, proxy logs, or network monitoring tools can extract valid JWT tokens from URL query strings. Once obtained, the attacker can authenticate as the victim user for up to 24 hours (the token expiry period).

VULNERABILITY #5
--------------------------------------------------------------------------------
CWE ID: CWE-22 (Improper Limitation of a Pathname to a Restricted Directory)
Vulnerability Name: Potential Path Traversal via Database-Stored File Path
Severity: Medium
File: src/main.cpp
Lines: 464-474
Description: The streaming endpoint retrieves the file_path directly from the Firebase database and uses it to read and serve a file from the filesystem without validating that the path is within the expected uploads directory. If an attacker were to manipulate the database record, they could set file_path to an arbitrary filesystem path.
Exploitation: If an attacker gains write access to the Firebase database (through a separate vulnerability, compromised credentials, or Firebase misconfiguration), they could modify a song's file_path to point to any file on the server (e.g., "firebase-credentials.json" or application source files), leading to arbitrary file read.
