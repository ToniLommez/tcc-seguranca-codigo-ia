VULNERABILITY #1
--------------------------------------------------------------------------------
CWE-798 (Use of Hard-Coded Credentials)
Severity: Critical
File: cmd/server/main.go, Line 17
Description: A hardcoded filesystem path to a Firebase service account key file
is embedded directly in the source code as a fallback default:
  credPath = `C:\Users\tonil\Desktop\tcc\lead-manager-54595-firebase-adminsdk-fbsvc-375221693a.json`
This path reveals the Firebase project name ("lead-manager-54595") and the
service account key identifier ("375221693a"). If this code is committed to a
repository (public or shared), attackers gain knowledge of the exact credential
file location and the Firebase project identity, facilitating targeted attacks.
Exploitation: An attacker who gains read access to the source code (e.g., via a
public repository) learns the exact Firebase project and credential file path.
Combined with any file-read vulnerability or access to the developer machine,
the full service account key could be exfiltrated, granting full Firestore
database access.

VULNERABILITY #2
--------------------------------------------------------------------------------
CWE-798 (Use of Hard-Coded Credentials)
Severity: Critical
File: internal/auth/jwt.go, Lines 17-21
Description: The JWT signing secret has a hardcoded fallback value:
  secret = "lead-manager-jwt-secret-change-in-production"
If the JWT_SECRET environment variable is not set (which is the default), this

Exploitation: An attacker crafts a JWT with an arbitrary user_id and email,

VULNERABILITY #3
--------------------------------------------------------------------------------
CWE-942 (Overly Permissive Cross-Origin Resource Sharing)
Severity: High
File: internal/middleware/auth.go, Lines 43-44
Description: The CORS middleware sets Access-Control-Allow-Origin to "*",
allowing any website on the internet to make authenticated cross-origin requests
to the API. Combined with the fact that Authorization headers are also allowed,
any malicious website can make API calls on behalf of a logged-in user if the
token is accessible via JavaScript.
Exploitation: An attacker hosts a malicious webpage that makes AJAX requests to
the Lead Manager API. If a user visits the page while having their JWT token
stored in localStorage (common for SPAs), the attacker's JavaScript can read the
token and make authenticated API requests to steal or modify the user's leads.

VULNERABILITY #8
--------------------------------------------------------------------------------
CWE-352 (Cross-Site Request Forgery)
Severity: Medium
File: cmd/server/main.go (entire routing), internal/middleware/auth.go
Description: The application has no CSRF protection. While it uses JWT Bearer
tokens (which provide some CSRF resistance since they must be explicitly
attached), the tokenOrBearerAuth function on lines 140-146 of main.go accepts
tokens via query parameters, which can be embedded in links. State-changing
operations (DELETE, PUT, POST) on lead resources have no CSRF token validation.
Exploitation: An attacker could craft a malicious link with a stolen or known
token in the query string that triggers a state-changing export operation. If
tokens are stored in cookies in any future iteration, all state-changing
endpoints become vulnerable to CSRF.

VULNERABILITY #10
